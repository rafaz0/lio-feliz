## 1. Objetivo

Esta Product Interpretation (PI-005) estabelece a arquitetura oficial da Application Layer do Lio Feliz.

Seu objetivo é definir como a camada de aplicação orquestra o domínio patrimonial (PI-004), coordena transações, expõe portas para infraestrutura e serve como ponto de entrada para todos os casos de uso do sistema.

A Application Layer é uma camada fina de orquestração. Ela não contém regras de negócio — essas pertencem exclusivamente ao domínio (PI-004). Ela não implementa persistência, conectividade ou interface — essas responsabilidades pertencem à Infrastructure e ao Frontend.

Esta PI estabelece os contratos arquiteturais entre a Application Layer e as demais camadas, garantindo que o núcleo do domínio permaneça puro, testável e independente de tecnologia.

---

### 1.1 Motivação

O projeto concluiu o domínio patrimonial (EWO-002) com 362 testes, 12 Decisões Arquiteturais e 13 Invariantes implementadas. O domínio é puro — zero dependências de infraestrutura.

Para que o sistema se torne utilizável, é necessária uma camada que:

- orquestre as operações do domínio em fluxos completos (casos de uso);
- coordene transações entre múltiplas operações do domínio;
- exponha interfaces (ports) que a Infrastructure possa implementar;
- traduza requisições externas em comandos compreensíveis pelo domínio;
- traduza respostas do domínio em DTOs para as camadas superiores;
- gerencie autorização, validação de entrada e cross-cutting concerns.

Sem esta definição arquitetural, cada caso de uso seria implementado de forma ad hoc, gerando acoplamento entre a orquestração e os detalhes de infraestrutura.

---

### 1.2 Escopo

Esta PI contempla:

- princípios arquiteturais da Application Layer;
- linguagem ubíqua da camada de aplicação;
- arquitetura de Commands e Queries (CQRS);
- mapeamento dos 14 Casos de Uso (UC-001 a UC-014) para Application Services;
- definição de Ports (interfaces para Infrastructure);
- definição de DTOs;
- tratamento de Domain Events na Application Layer;
- gerenciamento de transações e Unit of Work;
- tratamento de erros da camada de aplicação;
- invariantes arquiteturais;
- dependências permitidas e proibidas;
- rastreabilidade com FRs, UCs e NFRs.

---

### 1.3 Fora do Escopo

Esta PI não define:

- regras de negócio — pertencem ao domínio (PI-004);
- modelo canônico de investimentos — pertence à PI-002;
- fluxo de eventos operacionais — pertence à PI-003;
- interpretação de dados externos — pertence à PI-001;
- persistência, banco de dados, ORM — pertencem à Infrastructure;
- APIs HTTP, GraphQL, gRPC — pertencem à Infrastructure;

- sincronização com provedores externos — pertence à Infrastructure;
- filas, mensageria, eventos de infraestrutura — pertencem à Infrastructure;
- caching, performance otimizations — pertencem à Infrastructure.

Essas responsabilidades serão definidas em PIs próprias ou nas EWOs de Infrastructure e Frontend.

## 2. Posicionamento Arquitetural

A Application Layer ocupa a camada de orquestração entre o Domínio e a Infrastructure.

```
Frontend (UI, dashboards, formulários)
    ↓
Application Layer (serviços, orquestração, ports)
    ↓
Domínio (regras de negócio, eventos, projeções)
    ↓
Infrastructure (repositórios, APIs, persistência)
```

A Application Layer:
- depende do Domínio (PI-004) — invoca suas entidades, Aggregate Roots e serviços de domínio;
- é dependida pela Infrastructure — a Infrastructure implementa os Ports definidos aqui;
- é consumida pelo Frontend — o Frontend chama os Application Services ou envia Commands/Queries.

Ela nunca depende de Infrastructure ou Frontend.

A Application Layer conhece o Domínio. A Infrastructure conhece a Application Layer. O Frontend conhece a Application Layer.

Essa direção de dependências é obrigatória e será validada em toda Engineering Review.

---

## 3. Princípios Arquiteturais

A Application Layer deverá respeitar permanentemente os seguintes princípios.

### PA-005-001 — Orquestração, não Implementação

A Application Layer orquestra operações do domínio. Ela nunca implementa regras de negócio. Toda regra de negócio pertence ao domínio (PI-004).

### PA-005-002 — Ports como Único Ponto de Acoplamento

A Application Layer define Ports (interfaces) para toda comunicação com Infrastructure. Nenhum Application Service depende diretamente de implementações concretas de persistência, API externa ou serviço de infraestrutura.

### PA-005-003 — Commands e Queries como Contrato de Entrada

Toda requisição que ingressa na Application Layer é representada por um Command (operações que alteram estado) ou uma Query (operações de consulta). Nenhum Application Service recebe parâmetros genéricos ou objetos de transporte não tipados.

### PA-005-004 — DTOs como Contrato de Saída

Toda resposta produzida pela Application Layer para camadas superiores é representada por um DTO (Data Transfer Object). Nenhuma entidade do domínio atravessa a fronteira da Application Layer.

### PA-005-005 — Transação por Caso de Uso

Cada chamada a um Application Service define um limite transacional. O Application Service coordena o Unit of Work que garante consistência entre múltiplas operações do domínio dentro de um mesmo caso de uso.

### PA-005-006 — Responsabilidade Única por Service

Cada Application Service implementa exatamente um caso de uso. Não existem serviços genéricos que acumulam múltiplas responsabilidades.

### PA-005-007 — Validação de Entrada na Porta de Entrada

A validação de formato, tipos, campos obrigatórios e regras de apresentação dos dados de entrada ocorre na fronteira da Application Layer (antes do domínio). A validação de regras de negócio permanece no domínio.

### PA-005-008 — Domain Events como Mecanismo de Reação

A Application Layer escuta Domain Events produzidos pelo domínio e dispara reações (notificações, sincronizações, atualizações de projeções). Ela nunca modifica o passado — apenas reage a fatos consumados.

## 4. Linguagem Ubíqua da Application Layer

Os conceitos a seguir constituem a linguagem ubíqua oficial da Application Layer.

### 4.1 Application Service

Application Service representa a unidade de orquestração de um caso de uso. Ele coordena uma ou mais operações do domínio, gerencia transações e produz DTOs de resposta.

Exemplos: `RegistrarOperacaoService`, `ConsultarPatrimonioService`.

### 4.2 Command

Command representa uma intenção de alterar o estado do sistema. Ele é imutável, nomeado de acordo com a ação e contém todos os dados necessários para executar a operação.

Exemplos: `RegistrarCompraCommand`, `ImportarCarteiraCommand`, `ConfigurarEstrategiaCommand`.

### 4.3 Query

Query representa uma intenção de consultar o estado do sistema. Ela é imutável, nomeada de acordo com a informação desejada e contém todos os filtros necessários.

Exemplos: `ObterPatrimonioQuery`, `ConsultarPosicaoQuery`, `ObterHistoricoQuery`.

### 4.4 DTO (Data Transfer Object)

DTO representa a resposta produzida por um Application Service. Ele é imutável e não expõe entidades do domínio. DTOs são produzidos apenas pela Application Layer.

### 4.5 Port

Port representa uma interface que a Application Layer define para que a Infrastructure implemente. Ports garantem inversão de dependência — a Application Layer não conhece implementações concretas.

Exemplos: `IPortfolioRepository`, `IAssetRepository`, `IUnitOfWork`.

### 4.6 Unit of Work

Unit of Work representa o limite transacional de um caso de uso. Ele garante que múltiplas operações sejam persistidas atomicamente ou revertidas em caso de falha.

### 4.7 Application Error

Application Error representa um erro da camada de aplicação. Ele não representa um erro de domínio (Domain Error) — representa falhas de validação de entrada, autorização, concorrência ou recursos não encontrados.

## 5. Arquitetura de Commands e Queries (CQRS)

A Application Layer adota CQRS como padrão arquitetural de entrada e saída.

Esta decisão não implica infraestrutura separada para leitura e escrita — é uma decisão de modelo, não de deployment.

### 5.1 Commands

Commands alteram estado. Eles são processados por Application Services que orquestram o domínio e coordenam transações.

Características de um Command:
- nomeado no imperativo: `RegistrarCompraCommand`, `ImportarCarteiraCommand`;
- imutável após criação;
- contém todos os dados necessários para execução;
- produz um DTO de confirmação ou erro;
- nunca retorna entidades do domínio.

### 5.2 Queries

Queries consultam estado. Elas são processadas por Application Services que acessam projeções ou leem dados sem modificar o domínio.

Características de uma Query:
- nomeada no formato `Obter[X]Query` ou `Consultar[X]Query`;
- imutável após criação;
- contém filtros e parâmetros de consulta;
- produz um DTO de resposta;
- nunca altera o domínio.

### 5.3 Dispatcher

A Application Layer expõe um mecanismo de dispatcher que encaminha Commands e Queries para os Application Services correspondentes.

O dispatcher:
- recebe um Command ou Query;
- valida a entrada estruturalmente;
- localiza o Application Service responsável;
- executa o serviço;
- retorna o DTO de resposta ou Application Error.

### 5.4 Processamento de Commands

```
Command → Dispatcher → Validação de Entrada → Application Service → Domínio (múltiplas operações) → Unit of Work → DTO
```

### 5.5 Processamento de Queries

```
Query → Dispatcher → Application Service → Projeção / Leitura → DTO
```

Queries não passam por Unit of Work — elas apenas leem dados.

### 6.1 Estrutura de um Application Service

Cada Application Service implementa a interface `IApplicationService<TCommand, TDto>` ou equivalente, contendo:

- método único de execução (ex.: `ExecuteAsync`);
- dependência exclusiva de Ports (interfaces) — nunca de implementações concretas;
- orquestração de operações do domínio;
- coordenação do Unit of Work para Commands;
- produção de DTO de resposta.

Nenhum Application Service pode conter:
- regras de negócio (if/else que decide comportamento patrimonial);
- acesso direto a banco de dados;
- chamadas a APIs externas;
- lógica de apresentação.

### 6.2 Serviços de Command

#### 6.2.1 ImportarCarteiraService

**UC:** UC-002 | **FRs:** FR-022, FR-023, FR-024

Responsável por orquestrar a importação de dados de fonte externa (CSV, B3).

Fluxo:
1. Recebe `ImportarCarteiraCommand` com os dados brutos
2. Invoca a Interpretation Layer (PI-001) para converter dados em operações canônicas
3. Para cada operação válida, invoca o domínio para registrar Financial Events no Portfolio
4. Coordena o Unit of Work para persistência atômica
5. Retorna `ImportacaoRealizadaDto` com resumo do resultado

#### 6.2.2 RegistrarOperacaoService

**UC:** UC-003 | **FRs:** FR-001, FR-007, FR-009

Responsável por registrar operações financeiras manuais (compra, venda, eventos corporativos).

Fluxo:
1. Recebe `RegistrarOperacaoCommand` com tipo, ativo, quantidade, valor, data
2. Invoca o domínio para registrar o Financial Event no Portfolio
3. Coordena o Unit of Work
4. Retorna `OperacaoRegistradaDto`

#### 6.2.3 ConfigurarEstrategiaService

**UC:** UC-008 | **FRs:** FR-019, FR-025, FR-026, FR-027

Responsável por persistir as configurações de estratégia do usuário.

Fluxo:
1. Recebe `ConfigurarEstrategiaCommand` com percentuais, moeda, metas
2. Persiste as configurações via Port de repositório de configurações
3. Coordena o Unit of Work
4. Retorna `EstrategiaConfiguradaDto`

#### 6.2.4 GerenciarAssinaturaService

**UC:** UC-011 | **FRs:** FR-032, FR-033, FR-034

Responsável por gerenciar planos e assinaturas.

Nota: Este serviço orquestra regras de assinatura que pertencem a um contexto ainda não definido arquiteturalmente. Seu detalhamento será refinado quando a camada de Licensing & Feature Access for especificada.

Fluxo:
1. Recebe `GerenciarAssinaturaCommand` com plano, ação (ativar, cancelar, alterar)
2. Valida a transição de plano
3. Persiste via Port de repositório de assinaturas
4. Coordena o Unit of Work
5. Retorna `AssinaturaAtualizadaDto`

#### 6.2.5 SincronizarDadosService

**UC:** UC-012 | **FR:** FR-023

Responsável por executar sincronização automática com fontes externas.

Fluxo:
1. Recebe `SincronizarDadosCommand` com identificação da fonte
2. Invoca o adaptador correspondente via Port de gateway de dados
3. Para cada operação obtida, invoca Interpretation Layer e domínio
4. Coordena o Unit of Work para cada lote
5. Retorna `SincronizacaoRealizadaDto`

## 7. Ports

Ports são interfaces que a Application Layer define para que a Infrastructure implemente. Eles representam o único ponto de acoplamento entre Application Layer e Infrastructure.

### 7.1 Repositórios

#### IPortfolioRepository

```
- ObterPorId(portfolioId) → Portfolio?
- Salvar(Portfolio) → void
- ObterTodos(usuarioId) → Portfolio[]
```

Nota: `Salvar` persiste o Aggregate Root e todos os eventos não processados. O detalhamento da estratégia de persistência será definido na EWO-003.

#### IAssetRepository

```
- ObterPorId(assetId) → Asset?
- ObterPorTicker(ticker) → Asset?
- Salvar(Asset) → void
- Listar() → Asset[]
```

#### IConfigurationRepository

```
- ObterEstrategia(usuarioId) → EstrategiaConfiguracao?
- SalvarEstrategia(usuarioId, EstrategiaConfiguracao) → void
- ObterMetas(usuarioId) → MetaFinanceira[]
- SalvarMetas(usuarioId, MetaFinanceira[]) → void
```

#### ISubscriptionRepository

```
- ObterPlanoAtivo(usuarioId) → Assinatura?
- Salvar(Assinatura) → void
- ListarPlanosDisponiveis() → PlanoDto[]
```

### 7.2 Repositórios de Projeção

Projeções não são persistidas como fonte primária — são construídas a partir dos eventos. Entretanto, a Application Layer pode acessar projeções materializadas para consultas eficientes.

#### IProjectionRepository

```
- ObterPatrimonio(portfolioId) → PatrimonioProjection?
- ObterPosicao(portfolioId, assetId) → PosicaoProjection?
- ObterPosicoes(portfolioId) → PosicaoProjection[]
- ObterHistorico(portfolioId, periodo) → HistoricoProjection[]
- ObterProventos(portfolioId, filtro) → ProventoProjection[]
```

### 7.3 Gateways

Gateways representam portas para serviços externos.

#### IDataGateway

```
- ObterDadosImportacao(origem, parametros) → DadosImportacao
```

Utilizado por `SincronizarDadosService` e `ImportarCarteiraService` para obter dados de fontes externas.

**Nota arquitetural:** Este Port utiliza um contrato genérico que pode exigir especialização por origem de dados (B3, CSV, APIs futuras) durante a EWO-003, caso os parâmetros de cada fonte sejam substancialmente diferentes.

### 7.4 Unit of Work

#### IUnitOfWork

```
- IniciarTransacao() → void
- Commit() → void
- Rollback() → void
```

### 7.5 Domain Event Publisher

#### IDomainEventPublisher

```
- Publicar(DomainEvent) → void
- PublicarVarios(DomainEvent[]) → void
```

### 7.6 Notification

#### INotificationPort

```
- Notificar(usuarioId, título, mensagem) → void
- NotificarEmail(usuarioId, assunto, corpo) → void
```

Utilizado por handlers de Domain Events (Seção 9.2) para notificar usuários sobre eventos relevantes (importação concluída, sincronização realizada, etc.).

```
  erros: SincronizacaoErroDto[]
}
```

## 9. Domain Events na Application Layer

O domínio (PI-004) produz Domain Events como fatos imutáveis que representam algo relevante ocorrido. A Application Layer escuta esses eventos e coordena reações.

### 9.1 Fluxo

```
Domínio produz DomainEvent → IDomainEventPublisher → Application Layer → Handlers → Reações
```

### 9.2 Handlers

Handlers são componentes da Application Layer que reagem a Domain Events.

Exemplos:

- `FinancialEventRegisteredHandler`: ao registrar um evento financeiro, atualiza projeções de posição e patrimônio
- `ImportacaoConcluidaHandler`: ao concluir uma importação, notifica o usuário (via Port de notificação)
- `AssinaturaAlteradaHandler`: ao alterar assinatura, atualiza feature flags

### 9.3 Reações Suportadas

A Application Layer coordena as seguintes reações a Domain Events:

- atualização de projeções materializadas;
- envio de notificações ao usuário;
- disparo de sincronizações complementares;
- registro de auditoria.

Nenhuma reação pode modificar o passado do domínio. Handlers apenas reagem a fatos consumados.

## 10. Gerenciamento de Transações

### 10.1 Limite Transacional

Cada chamada a um Application Service Command define um limite transacional.

```
Início da transação → Operações do domínio (1..N) → Commit ou Rollback
```

### 10.2 Unit of Work

O Unit of Work é iniciado pelo Application Service antes de orquestrar o domínio e finalizado após todas as operações.

Se qualquer operação falhar:
- o Unit of Work executa Rollback;
- nenhuma alteração parcial é persistida;
- o Application Service retorna um Application Error.

Se todas as operações forem bem-sucedidas:
- o Unit of Work executa Commit;
- Domain Events são publicados;
- o Application Service retorna o DTO de sucesso.

### 10.3 Queries

Queries não utilizam Unit of Work. Elas acessam dados de leitura sem coordenar transações.

### 10.4 Consistência Eventual

A Application Layer não exige consistência imediata entre aggregates em transações distintas. Cada transação garante consistência apenas dentro do seu próprio limite transacional.

## 11. Tratamento de Erros

### 11.1 Application Errors

A Application Layer define os seguintes tipos de erro:

| Erro | Causa | HTTP equivalente |
|------|-------|------------------|
| `ValidationError` | Dados de entrada inválidos | 400 |
| `NotFoundError` | Recurso não encontrado | 404 |
| `AuthorizationError` | Usuário sem permissão | 403 |
| `ConflictError` | Concorrência ou estado inválido | 409 |
| `DomainError` | Erro propagado do domínio | 422 |
| `InternalError` | Erro inesperado | 500 |

### 11.2 Propagação de Erros do Domínio

Quando o domínio retorna um Domain Error, a Application Layer:

1. captura o erro;
2. interrompe a transação (Rollback);
3. converte para `ApplicationError` do tipo `DomainError`;
4. sanitiza a mensagem — preserva a descrição semântica do erro sem expor detalhes internos de implementação do domínio (nomes de invariantes, valores intermediários, contexto de validação);
5. retorna para a camada superior com a mensagem sanitizada.

### 11.3 Erros de Infrastructure

Erros originados na Infrastructure (falha de banco, timeout de API, etc.) são capturados pela Application Layer como `InternalError`.

A Application Layer nunca expõe detalhes internos da Infrastructure para as camadas superiores.

## 12. Invariantes Arquiteturais

Os seguintes invariantes jamais poderão ser violados.

### IA-043 — Domínio Puro

A Application Layer jamais poderá conter regras de negócio do domínio patrimonial. Toda regra de negócio pertence ao domínio (PI-004).

### IA-044 — Inversão de Dependência

A Application Layer jamais poderá depender diretamente de implementações concretas de Infrastructure. Toda comunicação com Infrastructure ocorre exclusivamente via Ports.

### IA-045 — Entidades do Domínio Não Atravessam a Fronteira

Nenhuma entidade do domínio (Portfolio, Asset, FinancialEvent) poderá ser retornada diretamente para camadas superiores. Toda resposta deve ser um DTO.

### IA-046 — Um Command, uma Mutação

Cada Command processado pela Application Layer deve representar exatamente uma intenção de mutação. Commands genéricos que acumulam múltiplas intenções são proibidos.

### IA-047 — Transação Completa

Todo Command bem-sucedido deve ter sua transação concluída (Commit) antes de retornar à camada superior. Transações parciais são proibidas.

### IA-048 — Reações não Modificam o Passado

Handlers de Domain Events na Application Layer jamais poderão modificar eventos já registrados no domínio. Reações produzem apenas novos fatos ou efeitos colaterais.

### IA-049 — ValidationError não Propaga Erro de Domínio

Erros de validação de entrada (formato, tipo, campos obrigatórios) são detectados antes de qualquer operação do domínio. Nenhum ValidationError pode conter mensagens de erro originadas no domínio.

### IA-050 — Rastreabilidade de Erros

Todo Application Error deve conter informações suficientes para rastrear a causa do erro até a origem, sem expor detalhes internos de Infrastructure.

### 13.1 Dependências Permitidas

A Application Layer pode depender de:

- contratos da camada Licensing & Feature Access;

Até que essas camadas sejam especificadas, nenhuma implementação poderá depender delas.

## 15. Casos Limite

A implementação deverá prever, entre outros, os seguintes cenários.

### CL-005-001 — Command Inválido

Command com campos obrigatórios ausentes ou tipos incorretos. Retorna `ValidationError` sem acessar o domínio.

### CL-005-002 — Recurso Não Encontrado

Query solicita posição de ativo inexistente no Portfolio. Retorna `NotFoundError`.

### CL-005-003 — Violação de Regra de Domínio

Command tenta registrar evento que viola invariante do domínio. Retorna `DomainError` com a mensagem do domínio.

### CL-005-004 — Concorrência

Dois Commands simultâneos tentam modificar o mesmo Portfolio. O segundo detecta conflito e retorna `ConflictError`.

### CL-005-005 — Falha de Infrastructure

Banco de dados indisponível durante Commit. Retorna `InternalError` sem expor detalhes.

### CL-005-006 — Sincronização Parcial

Durante `SincronizarDadosService`, parte das operações é importada com sucesso e parte falha. O serviço registra sucessos e falhas, retornando `ImportacaoRealizadaDto` com ambos os resultados.

### CL-005-007 — Portfolio Vazio

Consulta de patrimônio em Portfolio sem eventos registrados. Retorna patrimônio zerado, não erro.

### CL-005-008 — Período sem Dados

Query de histórico ou rentabilidade para período sem movimentações. Retorna dados zerados, não erro.

### CL-005-009 — Usuário sem Plano Ativo

Command de operação financeira para usuário sem assinatura ativa (plano gratuito expirado). Retorna `AuthorizationError` quando aplicável.

## 16. Restrições Arquiteturais

### RA-005-001

Nenhum Application Service pode conter lógica condicional que decida comportamento patrimonial. Toda decisão de negócio pertence ao domínio.

### RA-005-002

Nenhum Application Service pode acessar banco de dados ou ORM diretamente. Toda persistência ocorre via Ports.

### RA-005-003

Nenhum Application Service pode chamar APIs externas diretamente. Toda comunicação externa ocorre via Ports (gateways).

### RA-005-004

Commands e Queries são imutáveis após criação. Nenhum mecanismo da Application Layer pode modificar um Command ou Query durante o processamento.

### RA-005-005

A Application Layer deve ser completamente testável sem Infrastructure real. Todo Port deve ter uma implementação falsa (mock, stub, fake) para testes unitários.

### RA-005-006

Novos Application Services devem ser adicionados sem modificar Application Services existentes. O dispatcher deve suportar registração de novos serviços sem alteração de código existente.

## 18. Impacto Arquitetural

A Application Layer estabelece a ponte entre o domínio puro e o mundo exterior.

A partir desta camada:
- o domínio permanece isolado de tecnologia — nenhuma dependência externa o contamina;
- a Infrastructure tem contratos claros (Ports) para implementar;
- o Frontend tem contratos claros (Commands, Queries, DTOs) para consumir;
- casos de uso são explicitamente mapeados e rastreáveis;
- transações têm limites definidos por service;
- erros têm categorias e comportamentos previsíveis.

Esta PI constitui o próximo pilar arquitetural após a conclusão do domínio patrimonial (EWO-002). Sem ela, a implementação da Infrastructure e do Frontend careceria de contratos estáveis.

## 19. Riscos Conhecidos

### RA-005-001 — Service Inchado

Um Application Service pode acumular responsabilidades além da orquestração, tornando-se um "god service".

**Mitigação:** Cada service implementa exatamente um UC. Engineering Reviews devem validar esse limite.

### RA-005-002 — Port Muito Genérico

Ports podem ser definidos de forma genérica demais, perdendo a semântica do domínio.

**Mitigação:** Ports devem ser nomeados por entidade de domínio (ex.: `IPortfolioRepository`, não `IRepository<T>`).

### RA-005-003 — Validação Duplicada

Validação de entrada na Application Layer pode duplicar validação de domínio.

**Mitigação:** Application Layer valida formato e presença (o que o domínio não precisa saber). Domínio valida regras de negócio (o que a Application Layer não deve conhecer).

### RA-005-004 — Acoplamento com Framework

O dispatcher de Commands/Queries pode criar dependência de um framework específico.

**Mitigação:** O dispatcher é um contrato da Application Layer. Sua implementação concreta pertence à Infrastructure.

### RA-005-005 — Transações Longas

Commands que processam múltiplas operações (ex.: importação de centenas de eventos) podem manter transações abertas por tempo excessivo.

**Mitigação:** Serviços com volume elevado (ex.: `SincronizarDadosService`) devem processar em lotes, com transações independentes por lote.
