## 1. Objetivo

Esta Product Interpretation (PI-008) estabelece a arquitetura para a expansão do domínio do Lio Feliz, completando as regras de negócio e anexos técnicos pendentes, e definindo como novos módulos de domínio estendem as 4 camadas congeladas (Core, Application, Infrastructure, Presentation) sem modificá-las.

Seu objetivo é definir a arquitetura para:

- os 8 novos módulos de Regras de Negócio (06 a 13);
- os 5 novos Anexos Técnicos pendentes;
- os novos Commands, Queries, DTOs, Ports e Handlers na Application Layer;
- os novos Adapters na Infrastructure Layer;
- as novas Features na Presentation Layer.

Toda evolução respeita as Frozen Baselines existentes e ocorre exclusivamente por extensão, nunca por modificação.

---

## 2. Princípios Arquiteturais

### PA-008-001 — Extensão, Nunca Modificação

Nenhuma camada congelada (Core Foundation, Application Layer, Infrastructure Layer, Presentation Layer) pode ser modificada. Toda evolução ocorre por adição de novos arquivos, novos contratos e novos componentes dentro da estrutura existente.

### PA-008-002 — Um Módulo por Domínio

Cada novo módulo de Regra de Negócio (06 a 13) é um módulo independente dentro de `src/core/domain/`, com suas próprias entidades, value objects, regras e serviços de domínio. Módulos compartilham apenas conceitos fundamentais já definidos nas regras globais (00_GLOBAL_RULES.md).

### PA-008-003 — Contratos Públicos por Módulo

Cada novo módulo expõe seus próprios Commands, Queries e DTOs na Application Layer. Commands e Queries seguem a nomenclatura padronizada: `<Acao><Modulo>Command` / `<Acao><Modulo>Query`.

### PA-008-004 — Feature por Módulo

Cada novo módulo equivale a uma Feature na Presentation Layer. Features seguem a estrutura estabelecida em PI-007: components, hooks, viewmodels, tests dentro de `src/presentation/features/<feature-name>/`.

### PA-008-005 — Composição Root Extensível

O Composition Root (`src/integrations/dispatcher/presentation-dispatcher.ts`) é extensível por adição: novos handlers são registrados em novos blocos condicionais, sem remover ou alterar registros existentes.

### PA-008-006 — Documentação como Parte da Arquitetura

Cada novo módulo exige a criação do documento de Regra de Negócio correspondente em `docs/06_BUSINESS_RULES/`, do(s) Anexo(s) Técnico(s) em `docs/07_TECHNICAL_ANNEXES/`, e a atualização do índice.

## 4. Arquitetura por Camada

### 4.1 Core Domain (Extensão)

Cada novo módulo adiciona ao Core:

**Entidades de Domínio**

Novas entidades por módulo, seguindo o padrão estabelecido em PI-004:

| Módulo          | Entidades Previstas                                       |
| --------------- | --------------------------------------------------------- |
| Rebalanceamento | `RebalancingOrder`, `AllocationTarget`, `AllocationDrift` |
| Metas           | `FinancialGoal`, `GoalProgress`, `GoalContribution`       |
| Impostos        | `TaxStatement`, `TaxLot`, `TaxEvent`                      |
| Renda Fixa      | `FixedIncomeAsset`, `Coupon`, `AmortizationSchedule`      |
| Internacional   | `ForeignAsset`, `CurrencyConversion`, `ExchangeRate`      |
| Import/Export   | `ImportJob`, `ExportJob`, `ImportMapping`                 |
| Integrações     | `IntegrationConfig`, `SyncLog`, `ConnectionStatus`        |
| Relatórios      | `ReportTemplate`, `ReportExecution`, `ReportSchedule`     |

**Regras de Domínio**

Cada módulo implementa suas regras de negócio conforme documentado no respectivo documento de Regras de Negócio (`06_BUSINESS_RULES/`).

**Serviços de Domínio**

Serviços de domínio isolam lógicas complexas, especialmente:

- `RebalancingService` — Cálculo de alocação-alvo vs. alocação atual
- `TaxCalculationService` — Cálculo de IR sobre operações e proventos
- `CurrencyConversionService` — Conversão entre moedas com taxa de câmbio
- `GoalProgressService` — Cálculo de progresso de metas financeiras

### 4.4 Presentation Layer (Extensão)

Cada novo módulo equivale a uma Feature na Presentation Layer, seguindo a estrutura estabelecida em PI-007:

```
src/presentation/features/
  rebalanceamento/
  goals/
  tax/
  fixed-income/
  international/
  import-export/
  integrations/
  reports/
```

Cada Feature contém:

- `components/` — Componentes React
- `hooks/` — Hooks com TanStack Query + Dispatcher
- `viewmodels/` — ViewModels e mappers
- `tests/` — Testes

O Composition Root (`presentation-dispatcher.ts`) é estendido com novos blocos de registro condicional para cada novo Command/Query.

## 6. Restrições

### R-001 — Frozen Baselines

Nenhuma camada congelada (Core Foundation, Application Layer, Infrastructure Layer, Presentation Layer) pode ser modificada. Toda evolução ocorre exclusivamente por extensão.

### R-002 — Dependência entre Módulos

Módulos podem depender de módulos anteriores (ex.: Impostos depende de Operações e Proventos) mas nunca criam dependência circular. A ordem de implementação segue a priorização definida na Seção 5.1.

### R-003 — Contratos Públicos

Commands e Queries de novos módulos seguem a nomenclatura e estrutura definidas em PI-005. Nenhum contrato existente é alterado.

### R-004 — Composition Root

O Composition Root (`presentation-dispatcher.ts`) é estendido exclusivamente por adição de novos blocos de registro. Nenhum registro existente é removido ou alterado.

### R-005 — Architecture Guard

O Architecture Guard (ESLint override `src/presentation/**`) permanece ativo para todos os novos módulos da Presentation Layer. Nenhum novo módulo pode importar Core, Infrastructure ou Supabase diretamente.

### R-006 — Feature-First Obrigatório

Todos os novos módulos da Presentation Layer seguem a estrutura Feature-First definida em PI-007. Não é permitida a criação de módulos fora de `src/presentation/features/`.

### R-007 — Testes Obrigatórios

Cada novo módulo requer testes unitários (Vitest) e testes de arquitetura (R-10) antes da aprovação. A cobertura mínima é de 90% para regras de domínio e 80% para componentes de UI.
