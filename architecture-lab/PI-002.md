**Status:** Approved

> **Fonte Canônica de Engenharia:** Esta PI é a Fonte Canônica oficial do Modelo Canônico de Investimentos. Nenhuma implementação poderá divergir das definições aqui estabelecidas sem a criação de uma nova versão aprovada.

# 1. Objetivo

## 1.1 Finalidade

A PI-002 estabelece o Modelo Canônico de Investimentos do projeto Lio Feliz.

Seu objetivo é definir uma linguagem semântica única para representar o domínio de investimentos de forma independente de:

- provedores externos;
- mecanismos de persistência;
- interfaces de usuário;
- motores de negócio;
- detalhes de implementação.

O Modelo Canônico constitui a representação oficial do domínio financeiro dentro da arquitetura do sistema.

Todo componente interno deverá operar utilizando exclusivamente essa representação.

## 1.2 Motivação

As informações financeiras consumidas pelo sistema possuem origens heterogêneas.

Cada provedor utiliza:

- nomenclaturas próprias;
- identificadores distintos;
- estruturas específicas;
- convenções particulares;
- diferentes níveis de qualidade dos dados.

Permitir que essas representações atravessem livremente a arquitetura produz:

- acoplamento entre módulos;
- duplicação de regras;
- inconsistências semânticas;
- aumento da complexidade de manutenção;
- dificuldade de evolução.

O Modelo Canônico elimina esse problema ao estabelecer uma representação interna única e estável para todo o domínio de investimentos.

## 1.3 Escopo

Esta especificação define:

- a linguagem oficial do domínio;
- a ontologia dos investimentos;
- os contratos semânticos;
- os princípios arquiteturais;
- as identidades canônicas;
- os relacionamentos entre entidades;
- os invariantes arquiteturais.

## 1.4 Fora do Escopo

Esta PI não define:

- regras tributárias;
- cálculos financeiros;
- cálculo de patrimônio;
- cálculo de dividendos;
- cálculo de performance;
- sincronização com provedores;
- persistência;
- APIs;
- interfaces gráficas;
- banco de dados;
- estratégias de cache;
- protocolos de comunicação;
- detalhes de implementação.

Essas responsabilidades pertencem a componentes especializados que deverão utilizar o Modelo Canônico como fundamento arquitetural.

---

# 2. Posicionamento Arquitetural

O Modelo Canônico ocupa o núcleo semântico da arquitetura do Lio Feliz.

Todas as informações provenientes de fontes externas deverão ser interpretadas pela PI-001 — Interpretation Layer antes de ingressarem no Modelo Canônico.

Da mesma forma, todos os motores de negócio deverão consumir exclusivamente entidades canônicas.

```
Fontes Externas
        |
        ▼
PI-001 — Interpretation Layer
        |
        ▼
PI-002 — Canonical Investment Model
        |
        ├──────────────┐
        ▼              ▼
Portfolio Engine   Dividend Engine
        |              |
        └──────┬───────┘
               ▼
         Demais Engines
               |
               ▼
      Interface do Usuário
```

O fluxo de dependências é unidirecional.

Os consumidores dependem do Modelo Canônico.

O Modelo Canônico não depende de seus consumidores.

---

# 3. Responsabilidades Arquiteturais

Compete ao Modelo Canônico:

- definir a ontologia oficial do domínio financeiro;
- estabelecer a linguagem canônica utilizada por toda a arquitetura;
- definir contratos semânticos estáveis;
- estabelecer identidades canônicas;
- definir os relacionamentos entre entidades;
- preservar os invariantes arquiteturais do domínio;
- atuar como Canonical Source of Truth para todas as representações internas.

---

# 4. Responsabilidades Explicitamente Excluídas

O Modelo Canônico não deverá:

- interpretar dados externos;
- importar informações;
- consumir APIs;
- realizar sincronizações;
- persistir dados;
- gerar relatórios;
- calcular indicadores;
- aplicar regras tributárias;
- executar recomendações;
- realizar qualquer processamento de negócio.

Seu único propósito é representar corretamente o domínio.

---

# 5. Canonical Source of Truth

O Modelo Canônico constitui a única autoridade semântica do domínio financeiro do projeto.

Consequentemente:

Toda representação interna de investimentos deverá derivar exclusivamente desta especificação.

Nenhum componente poderá criar modelos paralelos do domínio.

Sempre que existir divergência entre:

- modelos externos;
- modelos de persistência;
- modelos de interface;
- modelos específicos de um Engine;

prevalecerá a definição estabelecida nesta PI.

O Modelo Canônico representa a linguagem oficial do ecossistema Lio Feliz.

---

# 6. Considerações Arquiteturais

O Modelo Canônico não é:

- um modelo de banco de dados;
- um contrato de API;
- um formato de serialização;
- um DTO;
- uma estrutura de persistência;
- uma implementação de software.

O Modelo Canônico é uma especificação arquitetural.

Sua responsabilidade consiste em definir o significado oficial das entidades do domínio.

Toda implementação futura deverá preservar essa semântica.

> **Engineering Note:** A PI-001 e a PI-002 possuem responsabilidades complementares. A PI-001 responde à pergunta: "Como o sistema interpreta o mundo externo?" A PI-002 responde à pergunta: "Como o sistema representa esse mundo internamente?" Essa separação constitui uma das principais decisões arquiteturais do Engineering N1.

# 7. Ontologia do Domínio

## 7.1 Objetivo

A Ontologia do Domínio estabelece os conceitos fundamentais utilizados para representar investimentos dentro do ecossistema Lio Feliz.

Ela define o que existe no domínio, independentemente de:

- implementação;
- persistência;
- fornecedor de dados;
- regras de negócio;
- interface do usuário.

Toda entidade definida nesta seção representa um conceito arquitetural estável.

## 7.2 Princípios Ontológicos

A ontologia do domínio deverá obedecer aos seguintes princípios:

**O domínio precede a implementação.**

As entidades são definidas por seu significado, nunca pela forma como serão implementadas.

**A semântica precede os atributos.**

Uma entidade existe porque representa um conceito do domínio. Os atributos apenas descrevem esse conceito. A existência da entidade jamais dependerá da existência de determinado campo.

**A identidade precede o estado.**

Cada entidade possui identidade própria. Seu estado poderá evoluir ao longo do tempo. Sua identidade permanecerá estável.

**O domínio é independente da infraestrutura.**

Nenhuma entidade poderá depender de banco de dados, API, framework, tecnologia ou fornecedor externo.

---

# 8. Linguagem Canônica

O Modelo Canônico estabelece a linguagem oficial utilizada por todos os componentes internos.

Consequentemente:

Todo módulo deverá comunicar-se utilizando exclusivamente entidades canônicas.

Não é permitido o compartilhamento direto de:

- modelos de APIs;
- DTOs externos;
- estruturas de persistência;
- objetos específicos de interface.

Toda comunicação deverá ocorrer através da linguagem definida nesta especificação.

---

# 9. Entidades Fundamentais

A ontologia do domínio é composta pelas seguintes entidades fundamentais.

## 9.1 Ativo Econômico

Representa o objeto econômico subjacente.

Exemplos:

- empresa;
- índice;
- título público;
- commodity;
- moeda;
- ativo digital;
- outro ativo passível de investimento.

O Ativo Econômico representa o conceito econômico. Não representa sua forma de negociação.

## 9.2 Instrumento Financeiro

Representa a forma pela qual um Ativo Econômico é disponibilizado ao mercado.

Exemplos:

- ação;
- ETF;
- FII;
- BDR;
- título;
- CDB;
- debênture;
- contrato;
- token;
- outros instrumentos financeiros.

Um Instrumento Financeiro referencia exatamente um Ativo Econômico.

Um Ativo Econômico poderá originar múltiplos Instrumentos Financeiros.

## 9.3 Carteira

Representa um agrupamento lógico de posições de investimento.

A Carteira constitui o contêiner oficial das posições pertencentes a um determinado contexto patrimonial.

A especificação não impõe restrições quanto ao significado desse contexto.

Exemplos:

- carteira principal;
- carteira internacional;
- carteira previdenciária;
- carteira empresarial;
- carteira familiar.

## 9.4 Posição

Representa a exposição patrimonial de uma Carteira em relação a um Instrumento Financeiro.

A Posição não descreve operações individuais.

Ela representa o estado consolidado decorrente da sequência de eventos registrados.

## 9.5 Evento

Representa qualquer ocorrência capaz de modificar o estado do domínio.

Exemplos:

- compra;
- venda;
- bonificação;
- grupamento;
- desdobramento;
- amortização;
- rendimento;
- subscrição;
- transferência;
- outros eventos financeiros.

Eventos representam fatos históricos.

Uma vez registrados, preservam sua identidade ao longo do tempo.

---

# 10. Relações Semânticas

As entidades mantêm as seguintes relações fundamentais.

```
Ativo Econômico
        |
        ▼
Instrumento Financeiro
        |
        ▼
Posição
        |
        ▼
Carteira

Evento ─────────► Posição
```

As relações definidas nesta especificação representam vínculos conceituais.

Elas não determinam a forma de implementação dessas relações.

---

# 11. Fronteiras da Ontologia

As seguintes entidades não pertencem ao Modelo Canônico:

- usuário;
- autenticação;
- autorização;
- corretoras;
- provedores de dados;
- bancos;
- APIs;
- infraestrutura;
- persistência;
- interface gráfica;
- licenciamento;
- configurações da aplicação.

Esses elementos poderão referenciar entidades canônicas, mas não fazem parte da ontologia do domínio financeiro.

---

# 12. Considerações Arquiteturais

A Ontologia do Domínio deverá permanecer estável ao longo da evolução do projeto.

Novas funcionalidades deverão reutilizar as entidades existentes sempre que possível.

Novas entidades somente poderão ser introduzidas quando representarem conceitos genuinamente inéditos do domínio e não puderem ser modeladas como especializações das entidades já definidas.

Essa diretriz visa preservar a estabilidade semântica do Modelo Canônico e minimizar a proliferação desnecessária de conceitos.

---

# 13. Identidade Canônica

## 13.1 Objetivo

Toda entidade pertencente ao Modelo Canônico deverá possuir uma identidade própria, independente de qualquer sistema externo.

A identidade canônica representa o mecanismo oficial utilizado pela arquitetura para reconhecer uma entidade ao longo de todo o seu ciclo de vida.

Essa identidade constitui um conceito arquitetural.

Ela não depende da forma como será implementada.

## 13.2 Independência da Identidade

A identidade de uma entidade nunca deverá depender de atributos provenientes de terceiros.

Consequentemente, identificadores como:

- ticker;
- ISIN;
- CUSIP;
- código B3;
- código ANBIMA;
- identificadores de provedores externos;
- nomes comerciais;

não constituem identidade arquitetural.

Esses elementos representam apenas identificadores de referência.

## 13.3 Estabilidade

Uma identidade canônica deverá ser:

- única;
- estável;
- permanente;
- imutável;
- independente do estado;
- independente do mercado;
- independente da infraestrutura.

Uma entidade poderá modificar seus atributos.

Jamais sua identidade.

---

# 14. Identificadores de Referência

Além da identidade canônica, uma entidade poderá possuir qualquer quantidade de identificadores externos.

Exemplos:

- ticker;
- ISIN;
- CNPJ;
- código B3;
- identificadores proprietários.

Esses identificadores poderão ser utilizados para:

- interpretação;
- sincronização;
- integração;
- importação;
- exportação.

Entretanto, nunca substituirão a identidade canônica definida pelo sistema.

---

# 15. Contratos Canônicos

## 15.1 Objetivo

Os contratos canônicos definem as responsabilidades semânticas de cada entidade do domínio.

Esses contratos representam compromissos arquiteturais.

Eles não representam contratos de implementação.

## 15.2 Contrato do Ativo Econômico

O Ativo Econômico deverá representar exclusivamente o objeto econômico subjacente.

Não deverá conter informações específicas de negociação.

Não deverá conhecer:

- carteiras;
- posições;
- eventos;
- usuários.

## 15.3 Contrato do Instrumento Financeiro

O Instrumento Financeiro deverá representar a forma negociável de um Ativo Econômico.

Deverá conhecer exclusivamente:

- sua identidade;
- o Ativo Econômico ao qual pertence;
- suas características próprias.

Jamais deverá conhecer:

- carteiras;
- posições;
- usuários.

## 15.4 Contrato da Carteira

A Carteira deverá representar exclusivamente um agrupamento lógico de posições.

Não deverá conhecer detalhes internos dos Instrumentos.

Não deverá modificar entidades pertencentes ao domínio financeiro.

## 15.5 Contrato da Posição

A Posição representa a exposição consolidada da Carteira a um Instrumento Financeiro.

Seu estado deverá ser consequência da sequência de Eventos associados.

A Posição nunca deverá redefinir informações pertencentes ao Instrumento.

## 15.6 Contrato do Evento

O Evento representa um fato ocorrido no domínio financeiro.

Sua responsabilidade consiste em registrar ocorrências capazes de alterar o estado do sistema.

Eventos preservam seu significado histórico.

Não representam estados.

Representam acontecimentos.

---

# 16. Invariantes Arquiteturais

Os invariantes definidos nesta seção deverão permanecer verdadeiros durante toda a evolução do projeto.

### IA-001 — A Identidade é Imutável

Uma identidade canônica nunca poderá ser alterada após sua criação.

### IA-002 — O Estado é Derivado

O estado de uma entidade deverá ser consequência dos fatos registrados no domínio. Jamais deverá constituir a única fonte de verdade.

### IA-003 — Independência de Fontes Externas

O Modelo Canônico jamais dependerá diretamente de APIs, provedores, corretoras, bancos ou formatos proprietários. Toda interação ocorrerá por intermédio da PI-001.

### IA-004 — Instrumentos não Conhecem Carteiras

Instrumentos Financeiros jamais deverão possuir dependência semântica de Carteiras.

### IA-005 — Posições não Modificam Instrumentos

A evolução de uma Posição jamais altera a definição do Instrumento Financeiro correspondente.

### IA-006 — Eventos Preservam o Histórico

Eventos representam fatos históricos. Seu significado deverá permanecer preservado ao longo do tempo. Correções deverão ocorrer mediante novos eventos ou mecanismos equivalentes definidos pela implementação.

### IA-007 — Ontologia precede Implementação

Nenhuma decisão de implementação poderá alterar os conceitos definidos pela Ontologia do Domínio.

### IA-008 — Especializações preservam Contratos

Novos tipos de Instrumentos Financeiros deverão estender o Modelo Canônico sem romper os contratos definidos nesta especificação.

---

# 17. Princípios Arquiteturais

O Modelo Canônico adota os seguintes princípios fundamentais.

### PA-006 — Separação entre Identidade, Estado e Histórico

Toda entidade deverá distinguir claramente quem ela é, qual seu estado atual e como chegou ao estado atual.

### PA-007 — Referências por Identidade

Relacionamentos entre entidades deverão ocorrer utilizando identidades canônicas. Jamais por atributos externos.

### PA-008 — Canonical Source of Truth

A definição desta especificação constitui a única autoridade semântica do domínio financeiro. Nenhum componente poderá estabelecer conceitos concorrentes.

### PA-010 — Independência Identitária

Toda identidade canônica deverá permanecer independente de infraestrutura, implementação, persistência, mercado ou fornecedor de dados.

---

# 18. Considerações Arquiteturais

Os contratos e invariantes definidos nesta seção representam compromissos permanentes da arquitetura.

Mudanças futuras deverão preservar esses princípios ou justificar formalmente, por meio de uma nova PI aprovada, qualquer alteração que afete a semântica do domínio.

---

# 19. Temporalidade

## 19.1 Objetivo

O Modelo Canônico deverá representar entidades que evoluem ao longo do tempo, preservando simultaneamente identidade, consistência semântica, histórico e rastreabilidade.

O tempo constitui uma dimensão arquitetural do domínio e não poderá ser ignorado por futuras implementações.

## 19.2 Evolução Temporal

Cada entidade poderá sofrer alterações em seu estado. Entretanto, sua identidade permanecerá estável, seus contratos permanecerão válidos e sua semântica permanecerá preservada.

Mudanças de estado jamais implicam criação de uma nova entidade conceitual.

## 19.3 Ciclo de Vida

Entidades poderão atravessar diferentes estados ao longo do tempo.

```
Criado → Ativo → Suspenso → Encerrado
```

O encerramento operacional de uma entidade não implica sua remoção do domínio. Entidades encerradas permanecem relevantes para reconstrução histórica, auditoria, rastreabilidade e consistência patrimonial.

## 19.4 Persistência Histórica

Eventos históricos deverão permanecer semanticamente preservados.

Implementações poderão utilizar projeções, snapshots, agregações ou caches. Entretanto, tais mecanismos não alteram a responsabilidade arquitetural definida nesta especificação.

---

# 20. Evolução do Modelo Canônico

O Modelo Canônico foi concebido para evoluir ao longo do tempo.

Essa evolução deverá preservar compatibilidade arquitetural, estabilidade semântica, identidade das entidades e contratos públicos.

## 20.1 Inclusão de Novas Entidades

Novas entidades somente poderão ser adicionadas quando representarem conceitos inéditos do domínio, não puderem ser representadas como especializações das entidades existentes e preservarem todos os contratos definidos nesta PI.

## 20.2 Especializações

Especializações poderão acrescentar novos comportamentos. Entretanto, jamais poderão alterar o significado das entidades fundamentais, contratos públicos ou invariantes arquiteturais.

---

# 21. Compatibilidade Arquitetural

Toda futura PI relacionada ao domínio financeiro deverá ser compatível com esta especificação.

# 22. Critérios de Aceite

**Ontologia**

**Linguagem**

- Vocabulário oficial definido.

**Contratos**

**Identidade**

**Invariantes**

- Compatibilidade com PI-001 validada.

**Implementação**
A especificação não deverá impor tecnologias, linguagens, bancos de dados, frameworks ou estruturas de persistência.

# 23. Riscos Arquiteturais

Os seguintes riscos deverão ser continuamente monitorados durante a evolução do projeto.

### RA-001 — Duplicação Semântica

Criação de modelos paralelos ao Modelo Canônico.

**Mitigação:** Utilização obrigatória da PI-002 como única referência semântica.

### RA-002 — Acoplamento Externo

Introdução de conceitos provenientes diretamente de APIs ou provedores.

**Mitigação:** Toda integração deverá ocorrer exclusivamente através da PI-001.

### RA-003 — Degradação da Ontologia

Criação de entidades redundantes.

**Mitigação:** Reutilização prioritária das entidades existentes.

### RA-004 — Violação de Contratos

Alteração dos contratos públicos sem revisão arquitetural.

**Mitigação:** Toda alteração estrutural deverá ser precedida por Proposal aprovada.
