# PI-001 — Interpretation Layer

**Documento:** PI-001.md

**Versão:** v1.0 (Approved)

**Status:** Approved

**Tipo:** Engineering Specification

**Última atualização:** 13/07/2026

---

> **Fonte Canônica de Engenharia:** Esta PI é a Fonte Canônica oficial da Interpretation Layer. Nenhuma implementação poderá divergir das definições aqui estabelecidas sem a criação de uma nova versão aprovada.

---

# 1. Objetivo

A Interpretation Layer é responsável por transformar informações provenientes de diferentes fontes de dados em uma representação interna única, padronizada e semanticamente consistente para utilização pelo ecossistema do Lio Feliz.

Esta camada constitui o primeiro ponto oficial de convergência de dados do sistema.

Seu objetivo é eliminar diferenças de nomenclatura, estrutura, origem e representação dos ativos financeiros, permitindo que todas as camadas superiores operem exclusivamente sobre contratos internos padronizados.

A Interpretation Layer não possui conhecimento sobre regras de negócio, cálculo patrimonial, estratégia de investimentos, recomendações ou persistência de dados.

Sua única responsabilidade consiste em interpretar informações externas e convertê-las para o modelo interno oficial do sistema.

# 2. Motivação Arquitetural

O projeto foi concebido para suportar múltiplas formas de entrada de dados, incluindo, mas não se limitando a:

- importação manual;
- integração com a B3;
- importação proveniente de plataformas terceiras;
- arquivos estruturados (CSV, planilhas e formatos equivalentes);
- APIs futuras;
- novos conectores que venham a ser desenvolvidos posteriormente.

Cada origem poderá representar um mesmo ativo de maneira distinta.

Sem uma camada de interpretação, todas as demais partes do sistema precisariam conhecer individualmente cada formato existente, criando forte acoplamento entre a lógica de negócio e as integrações.

A Interpretation Layer elimina esse problema ao estabelecer um único contrato interno de representação dos ativos.

Dessa forma, novas integrações podem ser adicionadas sem modificar o restante da arquitetura.

# 3. Problema que Resolve

Sem esta camada, o sistema apresentaria diversos problemas arquiteturais:

- duplicação de lógica de interpretação;
- dependência direta de provedores externos;
- aumento progressivo do acoplamento entre módulos;
- maior risco de regressão ao adicionar novas integrações;
- dificuldade de manutenção;
- inconsistência entre representações equivalentes de um mesmo ativo.

A existência da Interpretation Layer garante que todas as demais camadas recebam dados homogêneos, independentemente de sua origem.

# 4. Responsabilidade Exclusiva

A Interpretation Layer possui exatamente uma responsabilidade:

Interpretar informações externas e convertê-las para o modelo interno oficial do sistema.

Como consequência, ela:

**Deve:**

- interpretar dados recebidos;
- normalizar nomenclaturas;
- identificar o tipo do ativo;
- produzir uma representação interna padronizada;
- validar a conformidade estrutural mínima da entrada;
- comunicar inconsistências estruturais quando necessário.

**Não deve:**

- calcular patrimônio;
- calcular rentabilidade;
- realizar recomendações;
- executar regras tributárias;
- persistir dados;
- consultar preços de mercado;
- executar sincronizações;
- comunicar-se diretamente com interfaces do usuário;
- decidir estratégias de investimento.

Qualquer responsabilidade além da interpretação pertence às camadas superiores.

# 5. Escopo

A PI-001 contempla exclusivamente:

- definição da função arquitetural da Interpretation Layer;
- padronização do contrato interno de representação dos ativos;
- interpretação de múltiplas fontes compatíveis;
- desacoplamento entre integrações e regras de negócio;
- definição das responsabilidades da camada.

Esta PI não define detalhes de implementação, bibliotecas, linguagens de programação ou tecnologias específicas.

# 6. Fora do Escopo

Esta especificação não contempla:

- integração direta com provedores específicos;
- implementação da sincronização com a B3;
- importadores individuais;
- cálculo de carteira;
- Portfolio Engine;
- Trace Layer;
- Ledger Abstraction;
- CIR;
- interface gráfica;
- persistência em banco de dados;
- autenticação;
- autorização;
- infraestrutura.

Esses componentes serão definidos em PI próprias.

# 7. Critérios de Conclusão desta PI

Esta PI será considerada apta para receber o status **Approved** quando:

- todas as responsabilidades da Interpretation Layer estiverem claramente definidas;
- não existirem ambiguidades arquiteturais relevantes;
- o contrato da camada puder ser implementado sem decisões adicionais de arquitetura;
- o OpenCode puder gerar uma implementação apenas com base nesta especificação e nas interfaces por ela definidas.

# 8. Princípios Arquiteturais

A Interpretation Layer deverá respeitar permanentemente os seguintes princípios.

### PA-001 — Fonte Agnóstica

A camada nunca deverá depender de um provedor específico.

Toda informação recebida deverá ser tratada através de contratos internos, independentemente de sua origem.

Consequentemente, adicionar uma nova integração não deverá exigir alterações na lógica da camada.

### PA-002 — Modelo Canônico

Independentemente da origem dos dados, o resultado produzido pela camada deverá possuir sempre a mesma estrutura lógica.

As camadas superiores nunca deverão distinguir se um ativo foi obtido da B3, de um arquivo CSV, de uma importação manual ou de qualquer outra integração.

### PA-003 — Responsabilidade Única

A camada interpreta.

Ela não calcula.

Ela não decide.

Ela não recomenda.

Ela apenas converte uma representação externa para uma representação interna oficial.

### PA-004 — Determinismo

Para uma mesma entrada válida, a camada deverá produzir sempre a mesma representação interna.

Não poderá existir comportamento dependente de interface, usuário ou contexto externo.

### PA-005 — Extensibilidade

Novos provedores de dados deverão ser adicionados através da implementação de novos adaptadores.

Nunca através da modificação do contrato interno.

# 9. Fluxo Geral

O funcionamento lógico da Interpretation Layer será composto pelas seguintes etapas.

```
Fonte Externa
    ↓
Adapter
    ↓
Validação Estrutural
    ↓
Interpretação
    ↓
Normalização
    ↓
Modelo Canônico
    ↓
Camadas Superiores
```

Cada etapa possui responsabilidade exclusiva.

Nenhuma etapa poderá executar responsabilidades pertencentes às demais camadas da arquitetura.

# 10. Contrato de Entrada

A Interpretation Layer deverá receber informações provenientes de adaptadores compatíveis com o contrato oficial de entrada.

A camada não deverá conhecer como a informação foi obtida.

Ela deverá apenas receber um objeto compatível com o contrato estabelecido.

Esse contrato deverá fornecer, no mínimo:

- identificação do ativo;
- categoria ou tipo informado pela origem;
- origem dos dados (metadado);
- conjunto de atributos recebidos;
- informações suficientes para permitir a interpretação.

Campos adicionais poderão existir.

A camada nunca deverá depender deles para funcionar.

# 11. Contrato de Saída

Toda interpretação bem-sucedida deverá produzir um **Ativo Canônico**.

O Ativo Canônico representa a única forma oficial de representação de ativos dentro do Lio Feliz.

Independentemente da origem, todas as camadas superiores trabalharão exclusivamente com esse contrato.

O contrato deverá conter informações semanticamente padronizadas e independentes do provedor original.

Nenhuma camada superior poderá acessar diretamente estruturas específicas de integrações externas.

# 12. Garantias da Camada

Após uma interpretação válida, a Interpretation Layer garante que:

- o ativo está representado no formato interno oficial;
- nomenclaturas equivalentes foram normalizadas;
- diferenças de representação entre provedores foram eliminadas;
- a origem permanece disponível apenas como metadado;
- o restante do sistema poderá operar sem conhecer a fonte original.

# 13. Invariantes Arquiteturais

Os seguintes comportamentos jamais poderão ser violados.

### IA-001

Toda saída válida deverá obedecer ao Contrato Canônico.

### IA-002

A camada jamais poderá modificar regras de negócio.

### IA-003

A camada jamais poderá calcular patrimônio.

### IA-004

A camada jamais poderá executar recomendações.

### IA-005

Nenhuma integração poderá alterar o Modelo Canônico.

Novas integrações deverão adaptar-se ao contrato existente.

Nunca o contrário.

### IA-006

Falhas de interpretação nunca deverão produzir um Ativo Canônico parcialmente válido.

O resultado deverá indicar claramente sucesso ou falha, evitando estados inconsistentes.

# 14. Responsabilidades das Interfaces

Para evitar ambiguidades entre componentes da arquitetura, ficam definidas as seguintes responsabilidades:

- **Adaptadores:** Responsáveis por obter dados de uma fonte externa e convertê-los para o Contrato de Entrada da Interpretation Layer.
- **Interpretation Layer:** Responsável por interpretar e normalizar os dados recebidos, produzindo o Ativo Canônico.
- **Camadas Superiores:** Responsáveis por aplicar regras de negócio, cálculos, persistência, análises e demais funcionalidades do sistema.

# 15. Dependências Arquiteturais

A Interpretation Layer depende exclusivamente de contratos definidos pelas camadas inferiores responsáveis pela obtenção dos dados.

Ela não poderá possuir dependência direta de:

- provedores específicos;
- APIs externas;
- bancos de dados;
- interface do usuário;
- mecanismos de autenticação;
- mecanismos de licenciamento;
- regras de negócio;
- mecanismos de cálculo.

As únicas dependências permitidas são contratos formais de entrada e componentes internos estritamente necessários à interpretação.

Essa restrição garante baixo acoplamento e elevada reutilização da camada.

# 16. Regras Operacionais

A execução da Interpretation Layer deverá obedecer obrigatoriamente às seguintes regras.

### RO-001 — Independência da Origem

A interpretação deverá produzir exatamente o mesmo resultado lógico independentemente da origem dos dados.

Diferenças entre provedores deverão ser resolvidas exclusivamente durante a interpretação.

### RO-002 — Normalização Obrigatória

Toda representação equivalente deverá ser convertida para a nomenclatura oficial utilizada pelo modelo interno.

A padronização deverá ocorrer antes que qualquer camada superior receba o ativo.

### RO-003 — Não Persistência

A Interpretation Layer jamais poderá gravar informações permanentemente.

Persistência pertence exclusivamente às camadas responsáveis pelo armazenamento.

### RO-004 — Não Decisão

A camada jamais poderá decidir se um ativo deve ou não compor a carteira.

Ela apenas interpreta.

Toda decisão pertence às regras de negócio.

### RO-005 — Transparência

Sempre que possível, a origem do dado deverá permanecer registrada como metadado.

Essa informação poderá ser utilizada futuramente por mecanismos de auditoria, rastreabilidade e diagnóstico, sem interferir no modelo canônico.

### RO-006 — Compatibilidade Evolutiva

A inclusão de novos provedores de dados não poderá exigir modificações na lógica de interpretação existente.

Novos provedores deverão ser integrados exclusivamente por meio da implementação de adaptadores compatíveis com o contrato de entrada.

# 17. Tratamento de Falhas

A Interpretation Layer deverá tratar falhas de maneira determinística e previsível.

Uma falha de interpretação nunca poderá resultar em um Ativo Canônico parcialmente válido.

Toda operação deverá produzir um dos seguintes resultados:

- interpretação concluída com sucesso;
- interpretação rejeitada por inconsistência estrutural;
- interpretação rejeitada por ausência de informações obrigatórias;
- interpretação rejeitada por tipo de ativo não suportado;
- interpretação rejeitada por violação do contrato de entrada.

Sempre que ocorrer uma falha, a camada deverá fornecer informações suficientes para permitir diagnóstico técnico pelas camadas superiores.

A forma concreta de representação dessas falhas (objeto de resultado, resultado discriminado, exceção controlada ou mecanismo equivalente) será definida durante a implementação, desde que respeite este comportamento arquitetural.

# 18. Casos Limite

A implementação deverá prever, entre outros, os seguintes cenários.

- **CL-001** — Campos opcionais ausentes.
- **CL-002** — Campos obrigatórios inexistentes.
- **CL-003** — Representações diferentes para o mesmo ativo.
- **CL-004** — Nomenclaturas incompatíveis entre provedores.
- **CL-005** — Ativos ainda não suportados pelo sistema.
- **CL-006** — Metadados adicionais desconhecidos.
- **CL-007** — Evolução futura do contrato de entrada mantendo compatibilidade retroativa.

# 19. Restrições Arquiteturais

A implementação deverá respeitar obrigatoriamente as seguintes restrições.

### RA-001

A camada não poderá depender da interface gráfica.

### RA-002

A camada deverá ser reutilizável por qualquer módulo do sistema.

### RA-003

Nenhuma regra de negócio poderá ser implementada nesta camada.

### RA-004

Nenhuma decisão relacionada a planos, assinaturas, licenciamento ou autorização poderá ser implementada nesta camada.

Essas responsabilidades pertencem exclusivamente à futura camada de Licensing & Feature Access, preservando o princípio da responsabilidade única.

### RA-005

A camada deverá permanecer compatível com futuras integrações sem necessidade de alteração do seu contrato público.

A evolução do sistema deverá ocorrer preferencialmente pela adição de novos adaptadores e não pela modificação da camada.

# 20. Critérios de Aceite

A implementação da Interpretation Layer será considerada concluída quando todos os critérios abaixo forem atendidos.

### CA-001 — Independência de Provedor

A camada deverá interpretar corretamente dados provenientes de qualquer adaptador compatível com o Contrato de Entrada, sem depender de regras específicas de um provedor.

### CA-002 — Modelo Canônico

Toda interpretação bem-sucedida deverá produzir exclusivamente o Ativo Canônico definido pela arquitetura.

Não serão permitidas representações alternativas.

### CA-003 — Responsabilidade Única

A implementação não poderá conter lógica relacionada a:

- regras de negócio;
- cálculos financeiros;
- persistência;
- autenticação;
- autorização;
- licenciamento;
- interface gráfica;
- sincronização com provedores.

### CA-004 — Tratamento Determinístico

Entradas idênticas deverão produzir sempre o mesmo resultado.

Falhas deverão produzir sempre o mesmo comportamento previsto na especificação.

### CA-005 — Compatibilidade Evolutiva

A adição de um novo provedor de dados deverá ocorrer exclusivamente por meio de um novo adaptador compatível com o contrato estabelecido.

Não deverá ser necessária qualquer modificação na lógica da Interpretation Layer.

### CA-006 — Cobertura dos Casos Limite

Todos os casos limite definidos nesta PI deverão possuir comportamento explícito durante a implementação.

### CA-007 — Conformidade Arquitetural

A implementação deverá respeitar integralmente todos os contratos, princípios, invariantes e restrições definidos nesta especificação.

# 21. Impacto Arquitetural

A Interpretation Layer estabelece o primeiro ponto oficial de convergência entre o mundo externo e o modelo interno do Lio Feliz.

A partir desta camada:

- todas as representações de ativos passam a obedecer ao Modelo Canônico;
- as camadas superiores tornam-se independentes dos provedores de dados;
- novas integrações podem ser adicionadas sem alterar a lógica central do sistema;
- o acoplamento entre integrações e regras de negócio é significativamente reduzido.

Esta PI constitui um dos pilares arquiteturais do projeto.

# 22. Estratégia de Extensibilidade

A arquitetura da Interpretation Layer deverá permanecer estável ao longo da evolução do sistema.

Novas funcionalidades deverão ser introduzidas preferencialmente por:

- novos adaptadores;
- ampliação controlada dos contratos;
- novos tipos de ativos compatíveis com o Modelo Canônico;
- componentes auxiliares desacoplados.

A evolução da camada não deverá exigir alterações incompatíveis com implementações anteriores, preservando a retrocompatibilidade sempre que tecnicamente viável.

# 23. Diretrizes de Implementação

Embora esta PI não defina tecnologias específicas, toda implementação deverá observar os seguintes princípios.

### DI-001 — Baixo Acoplamento

Os componentes deverão possuir responsabilidades claramente definidas e dependências mínimas entre si.

### DI-002 — Alta Coesão

Cada componente deverá executar apenas uma função claramente identificável.

### DI-003 — Extensibilidade

A arquitetura deverá favorecer pontos de extensão em vez de modificações frequentes no núcleo da camada.

### DI-004 — Configuração acima de Código

Sempre que possível, comportamentos sujeitos à evolução deverão ser controlados por contratos, configurações ou mecanismos de extensão, reduzindo a necessidade de alterações estruturais no código.

### DI-005 — Manutenibilidade

A implementação deverá privilegiar soluções simples, legíveis e previsíveis.

Complexidade acidental deverá ser evitada.

### DI-006 — Compatibilidade com Evoluções Futuras

A implementação não deverá dificultar a futura introdução de:

- novos provedores de dados;
- novos tipos de ativos;
- mecanismos de auditoria;
- Trace Layer;
- Ledger Abstraction;
- Licensing & Feature Access Layer;
- demais componentes previstos na arquitetura.

# 24. Riscos Conhecidos

Os principais riscos arquiteturais associados à Interpretation Layer são:

- definição insuficiente do Modelo Canônico;
- crescimento descontrolado do contrato de entrada;
- acoplamento acidental com provedores específicos;
- introdução indevida de regras de negócio;
- utilização da camada para funções além da interpretação.

Esses riscos deverão ser monitorados durante as revisões de engenharia (ER).

# 25. Critérios para Aprovação

A PI poderá receber o status **Approved** quando:

- todas as seções estiverem completas;
- não existirem ambiguidades arquiteturais relevantes;
- os contratos estiverem definidos de forma suficiente para implementação;
- a revisão arquitetural for aprovada;
- a revisão de implementabilidade concluir que o OpenCode pode executar a implementação sem tomar decisões de arquitetura.
