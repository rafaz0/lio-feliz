## Objetivo

Materializar a **Onda 2 da PI-009** (Domain Expansion Ondas 2 & 3) através da implementação incremental dos 2 módulos:

1. **Renda Fixa (09)** — Novo módulo: regras de negócio, domínio, contratos, adapters e apresentação
2. **Internacional (10)** — Novo módulo: regras de negócio, anexo técnico de conversão de moeda, domínio, contratos, adapters e apresentação

Decisão arquitetural adotada (O2 resolvida): o **registro de operações** (compra/venda) de Renda Fixa e de ativos Internacionais **reutiliza o fluxo canônico `RegistrarOperacaoCommand` + `inferAssetType`** (reconhecimento dos tipos `fixed_income`, `stock_us`, `reit`, `etf_internacional`, `bdr`). Os comandos específicos de módulo cobrem apenas eventos/anexos próprios (`RegistrarCupomCommand`, `AtualizarTaxaCambioCommand`).

Esta Engineering Work Order transforma a especificação arquitetural aprovada em uma sequência rastreável de Slices de implementação, preservando integralmente as Frozen Baselines.

## Princípios Obrigatórios

1. **Extensão, Nunca Modificação** — Nenhum arquivo existente das camadas congeladas é alterado. Toda evolução ocorre por adição.
2. **Uma Slice = um incremento completo** — Compilável, testável, commitável e reversível.
3. **Testes dentro da Slice** — Nenhuma Slice é concluída sem seus testes.
4. **Auditoria obrigatória** — Sem auditoria da Slice anterior, não se inicia a próxima.
5. **Ordem técnica > ordem funcional** — BR + Domínio → App + Infra → Presentation.
6. **Fakes primeiro** — Implementações InMemory precedem as concretas (Supabase) para desacoplar testes.
7. **Zero quebra arquitetural** — PI-009 é a única fonte de arquitetura.
8. **Baseline Lock** — PI-009, ER-009 e EWO-007 entram em congelamento desde o Gate de Entrada.
9. **Fluxo canônico de operações (O2)** — Entrada de transações de Renda Fixa/Internacional via `RegistrarOperacaoCommand` + `inferAssetType`; comandos de módulo restritos a eventos/anexos.

## Estratégia Geral

### Ordem de Dependências

```
BR + Anexo Técnico (documentação)
        ↓
Core Domain (entidades, value objects, serviços)
        ↓
Application Contracts (Commands, Queries, DTOs, Ports)
        ↓
Infrastructure Adapters (Supabase + InMemory)
        ↓
Presentation Feature (components, hooks, viewmodels, tests)
```

Esta ordem se repete para cada módulo, respeitando a sequência: **Renda Fixa → Internacional**.

### Decisão O2 — Fluxo Canônico de Operações

O consolidador de carteira (EWO-002/005) opera sobre `Operation` + `inferAssetType`. Para evitar dupla fonte de verdade:

- A **entrada de compra/venda** de títulos de Renda Fixa e de ativos globais (ações US, REITs, ETFs internacionais, BDRs) usa o `RegistrarOperacaoCommand` já existente.
- O `inferAssetType` (PI-004 / agentes) deve reconhecer `fixed_income` e os tipos internacionais, alimentando a consolidação automaticamente.
- Os comandos de módulo são **específicos de evento/anexo**:
  - `RegistrarCupomCommand` — registra evento de cupom/amortização de um `FixedIncomeAsset`.
  - `AtualizarTaxaCambioCommand` — atualiza `ExchangeRate` usada por `CurrencyConversionService`.

### Dependências entre Módulos

```
09 — Renda Fixa (independente; consome apenas o fluxo canônico de operações)
        ↓
10 — Internacional (independente; consome fluxo canônico + CurrencyConversionService isolado)
```

### Estratégia de Testes

- **Testes unitários (Vitest):** obrigatórios, cobertura >90% do código novo
- **Testes de integração:** quando a Slice toca fronteiras (Ports, Dispatcher)
- **Testes de arquitetura:** verificar Dependency Rule e Architecture Guard para as novas features `fixed-income/` e `international/`
- **Fakes:** implementações InMemory utilizadas desde a Slice de Infra para viabilizar testes sem Supabase
