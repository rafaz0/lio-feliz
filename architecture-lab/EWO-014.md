## Objetivo

Materializar o Bloco B da PI-012 — Production Readiness — implementando a infraestrutura de engenharia necessária para operar o Lio Feliz em ambiente de produção.

Os pilares desta EWO são:

1. **CI/CD Pipeline** — Automação de build, teste e deploy via GitHub Actions
2. **Testes E2E** — Playwright para fluxos críticos com cobertura mínima
3. **Performance** — Code splitting, cache, lazy loading e otimização de assets
4. **Observabilidade** — Sentry para erros, logs estruturados e métricas

Nenhum Core Domain, regra de negócio ou contrato arquitetural é alterado.

---

## Escopo

### Inclui:

- Pipeline GitHub Actions com lint, typecheck, test, build e deploy
- Playwright test suite com fluxos críticos (auth, carteira, proventos, IRPF)
- Code splitting por rota/feature
- TanStack Query cache persistente + stale-while-revalidate
- Imagem/asset otimização via Vite + Cloudflare
- Sentry para erro tracking
- Logs estruturados via Cloudflare Workers
- Métricas de performance (Web Vitals)
- Documentação operacional

### Não contempla:

- Modificação de Core Domain, regras de negócio ou contratos
- Implementação de funcionalidades de usuário final
- Bloco C (Comercialização) — será coberto pela EWO-015

---

## Princípios Obrigatórios

1. **Infraestrutura como Código** — Toda configuração em arquivos versionados (`.github/`, `wrangler.toml`, `sentry.properties`)
2. **Zero quebra arquitetural** — Nenhum contrato público, regra de negócio ou camada congelada é alterada
3. **Testes automatizados** — CI falha se testes E2E não passarem
4. **Observabilidade first** — Toda nova feature deve incluir logging e monitoramento
5. **Performance mensurável** — Métricas antes/depois para cada otimização

---

## Estratégia de Implementação

A ordem segue a criticidade para produção:
1. CI/CD (bloqueante para deploys)
2. Observabilidade (captura erros desde o início)
3. Testes E2E (valida fluxos críticos)
4. Performance (otimizações incrementais)

## Planejamento Executivo das Slices

### Slice 1 — CI/CD Pipeline

| Campo | Valor |
|-------|-------|
| Número | 1 |
| Objetivo | Criar pipeline de CI/CD no GitHub Actions |
| Status | PENDENTE |

#### Arquivos a criar

- `.github/workflows/ci-cd.yml` — Pipeline completo:

```yaml
name: ci/cd
on: [push, pull_request]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run lint
      - run: npx tsc --noEmit
      - run: npx vitest run
      - run: npm run build

  deploy:
    needs: [quality]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run build
      - run: npx wrangler deploy
        env:
          CLOUDFLARE_API_TOKEN: \${{ secrets.CLOUDFLARE_API_TOKEN }}
```

#### Critérios de Aceite

- [ ] Pipeline executa lint + typecheck + tests + build
- [ ] Deploy automático para Cloudflare Workers em push para main
- [ ] Falha no quality gate bloqueia o merge

### Slice 4 — Performance

| Campo | Valor |
|-------|-------|
| Número | 4 |
| Objetivo | Otimizar performance da aplicação |
| Status | PENDENTE |

#### Otimizações

| Técnica | Onde | Como |
|---------|------|------|
| Code Splitting | Rotas pesadas (gráficos, tabelas) | `const Component = lazy(() => import("./Component"))` + Suspense |
| Query Cache | TanStack Query | `staleTime: 300_000`, `gcTime: 600_000`, persistência localStorage |
| Imagens | Assets estáticos | Vite `build.assetsInlineLimit`, Cloudflare Images |
| SSR Streaming | TanStack Start | Ativar `stream: true` no `createStart()` |

#### Métricas Alvo

| Métrica | Antes (est.) | Depois (meta) |
|---------|-------------|---------------|
| Bundle inicial | ~500 KB | < 300 KB |
| LCP | ~3s | < 1.5s |
| Req. API repetidas | 100% | 40% (com cache) |

#### Critérios de Aceite

- [ ] Code splitting em pelo menos 3 rotas pesadas
- [ ] TanStack Query com staleTime e gcTime configurados
- [ ] Bundle initial reduzido em 40% (verificar com `npx vite-bundle-analyzer`)
- [ ] SSR streaming ativado

## Riscos

| # | Risco | Prob. | Impacto | Mitigação |
|---|-------|-------|---------|-----------|
| 1 | Sentry expõe dados sensíveis em logs | Baixa | Alto | Configurar filtro de dados (denylist) no Sentry init |
| 2 | Playwright flaky em CI sem GPU | Média | Médio | Usar `xvfb-run` + retry 2x + headless mode |
| 3 | Code splitting causa regressão visual | Baixa | Baixo | Testes E2E nas rotas com lazy loading |
| 4 | CI/CD timeout com builds grandes | Baixa | Médio | Cache de `node_modules` entre runs |
