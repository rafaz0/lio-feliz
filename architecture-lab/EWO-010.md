Materializar a Onda 5 da PI-010 (Domain Enrichment & Investor Tooling) através da implementação incremental dos 2 módulos de Educação e Compliance:

1. **Educação (17)** — Novo módulo: glossário enriquecido, tooltips contextuais e learning paths. Sem anexo técnico.

## Princípios Obrigatórios

1. **Extensão, Nunca Modificação** — Nenhum arquivo existente das camadas congeladas é alterado. Toda evolução ocorre por adição.
2. **Uma Slice = um incremento completo** — Compilável, testável, commitável e reversível.
3. **Testes dentro da Slice** — Nenhuma Slice é concluída sem seus testes.
4. **Auditoria obrigatória** — Sem auditoria ChatGPT da Slice anterior, não se inicia a próxima.
5. **Ordem técnica > ordem funcional** — BR + Domínio → App + Infra → Presentation.
6. **Fakes primeiro** — Implementações InMemory precedem as concretas (Supabase) para desacoplar testes.
7. **Zero quebra arquitetural** — PI-010 é a única fonte de arquitetura.
8. **Baseline Lock** — PI-010, ER-010 e EWO-010 entram em congelamento desde o Gate de Entrada.
9. **Exportação assíncrona (R-014)** — Toda exportação regulatória é submetida via comando e materializada como job assíncrono. Nenhum arquivo regulatório é gerado inline na Presentation.
10. **Tooltips por composição, não modificação** — Módulo 17 injeta tooltips via `TooltipProvider` no Composition Root, nunca modificando componentes existentes (NC-010-003 resolvida).
11. **Scheduler compartilhado** — Módulo 18 reutiliza o mecanismo de scheduling do módulo 13 (Relatórios) para `AgendarExportacaoCommand`, evitando duplicação de infraestrutura (NC-010-005 resolvida).

O `Tooltip` entity do módulo 17 é usado exclusivamente como fonte de dados (texto, categoria, dificuldade) — a renderização é feita por um componente compartilhado injetado por contexto, não por alteração de componentes congelados. |

## Estratégia Geral

### Ordem de Dependências

```
BR + Anexo Técnico (documentação)
        ↓
Core Domain (entidades, value objects, serviços)
        ↓
Application Contracts (Commands, Queries, DTOs, Ports)
        ↓
Infrastructure Adapters (Supabase + InMemory)
        ↓
Presentation Feature (components, hooks, viewmodels, tests)
```

Esta ordem se repete para cada módulo, respeitando a sequência: **Educação (17) → Exportação Avançada (18)**.

### Dependências entre Módulos

```
17 — Educação (independente — não depende de módulos da Onda 4)
        ↓
18 — Exportação Avançada (depende de IReportRepository — módulo 13, já implementado na EWO-008)
```

### Estratégia de Testes

- **Testes unitários (Vitest):** obrigatórios, cobertura >90% do código novo
- **Testes de integração:** quando a Slice toca fronteiras (Ports, Dispatcher)
- **Testes de arquitetura:** verificar Dependency Rule e Architecture Guard para novos módulos (R-10 estendido, 0 violações)
- **Fakes:** implementações InMemory utilizadas desde a Slice de Infra para viabilizar testes sem Supabase
