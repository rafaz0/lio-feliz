## 1. Objetivo

Esta Engineering Review analisa a PI-005 (Application Layer do Lio Feliz) para:

- Validar a conformidade arquitetural, metodológica e documental da PI-005;
- Verificar a compatibilidade com PROJECT_BOOTSTRAP, AI_OPERATION_CHECKLIST, GOVs e IAs vigentes;
- Confirmar a aderência às PIs aprovadas (PI-001 a PI-004);
- Validar a rastreabilidade com 20_PROJECT_MAP.md e 21_FUNCTIONAL_SPECIFICATION.md;
- Verificar a consistência entre Use Cases, Application Services, Commands, Queries e Ports;
- Garantir a separação entre Domain, Application, Infrastructure e Presentation;
- Identificar não conformidades, ambiguidades ou ausências que impeçam a implementação via EWO-003;
- Emitir veredito formal sobre a prontidão da PI-005 para EWO-003.

A ER não altera arquitetura; apenas valida e detalha para implementação.

## 4. Não Conformidades Encontradas

### NC-005-001 — Ausência do Port de Notificação (Média)

**Local:** Seção 9.2 e Seção 7

**Descrição:** O handler `ImportacaoConcluidaHandler` (Seção 9.2) afirma notificar o usuário "via Port de notificação", mas a Seção 7 (Ports) não define nenhum `INotificationPort` ou equivalente. Isso cria uma dependência não especificada entre a Application Layer e a Infrastructure.

**Severidade:** Média — pode bloquear a implementação do handler caso o Port não exista.

**Correção proposta:** Adicionar à Seção 7 o Port:
```
INotificationPort
- Notificar(usuarioId, título, mensagem) → void
- NotificarEmail(usuarioId, assunto, corpo) → void
```

### NC-005-002 — Propagação de Mensagens do Domínio sem Sanitização (Menor)

**Local:** Seção 11.2 e CL-005-003

**Descrição:** A Seção 11.2 determina que a Application Layer deve "retornar para a camada superior com a mensagem original do domínio". Isso conflita parcialmente com IA-050 (Rastreabilidade de Erros) que proíbe expor detalhes internos. Mensagens de erro do domínio podem conter detalhes de implementação interna (nomes de invariantes, valores intermediários, contexto de validação) que não deveriam atravessar a fronteira.

**Severidade:** Menor — não bloqueia, mas pode causar vazamento de informação.

**Correção proposta:** Alterar Seção 11.2 para: "converte para `ApplicationError` do tipo `DomainError` com uma mensagem sanitizada — preserva a descrição semântica do erro sem expor detalhes internos de implementação do domínio."

### NC-005-003 — IDataGateway Excessivamente Genérico (Menor)

**Local:** Seção 7.3

**Descrição:** O Port `IDataGateway` possui um único método `ObterDadosImportacao(origem, parametros)` que trata todas as fontes de dados (CSV, B3, APIs futuras) com o mesmo contrato genérico. Diferentes fontes podem exigir parâmetros radicalmente diferentes (ex.: B3 requer autenticação, CSV requer stream de arquivo). O contrato genérico pode forçar type casting ou parâmetros opcionais em excesso.

**Severidade:** Menor — não bloqueia, mas requer atenção na EWO-003.

**Correção proposta:** Adicionar nota na Seção 7.3 recomendando que a EWO-003 avalie se `IDataGateway` deve ser especializado por origem ou se o contrato genérico é suficiente.

### NC-005-004 — Ambiguidade em IA-046 para Commands Compostos (Observação)

**Local:** IA-046, Seção 6.2.1

**Descrição:** IA-046 determina "Um Command, uma Mutação", mas `ImportarCarteiraCommand` representa uma única intenção ("importar arquivo") que pode resultar em múltiplas mutações no domínio (dezenas de Financial Events). Isso não viola a invariante (a intenção é única), mas pode causar interpretação incorreta.

**Severidade:** Observação — não requer alteração na PI.

**Correção proposta:** Nenhuma. A interpretação correta é que um Command = uma intenção de usuário, não uma operação atômica do domínio. Manter como está.
