## 1. Objetivo

Esta Product Interpretation (PI-013) estabelece a arquitetura para a maturidade da plataforma Lio Feliz, levando o produto do estado "completo em engenharia" para "pronto para produção real com excelência de experiência do usuário".

Seus objetivos são:

- preparar a plataforma para deploy real em produção, resolvendo pendências de infraestrutura e secrets;
- implementar as features de maior valor percebido ainda pendentes no backlog do produto;
- elevar a qualidade da experiência do usuário com responsividade mobile, performance e onboarding;
- iniciar a camada de inteligência e insights baseados nos dados patrimoniais do usuário;
- estabelecer a arquitetura de Licensing & Feature Access (BK-006).

Toda evolução respeita as Frozen Baselines (PI-001 a PI-007) e as PIs anteriores (PI-008 a PI-012). Nenhum port existente é modificado. Nenhuma regra de negócio é alterada. Nenhum Core Domain é tocado.

## 4. Arquitetura

### 4.1 Bloco A — Production Deployment

#### 4.1.1 Princípios

- **Zero-downtime deployment:** a migração para o ambiente de produção não pode interromper o acesso dos usuários.
- **Secrets seguros:** todas as credenciais devem ser gerenciadas via GitHub Secrets ou Cloudflare Secrets, nunca no código.
- **Observabilidade desde o primeiro dia:** Sentry e logs ativos antes do primeiro deploy.
- **Rollback rápido:** o pipeline deve permitir rollback para a versão anterior em menos de 5 minutos.

#### 4.1.2 Atividades Previstas

| Atividade | Descrição |
|-----------|-----------|
| Escolha da plataforma | Vercel (recomendado para TanStack Start) ou Cloudflare Workers |
| Configuração de secrets | Stripe secret key, Supabase URL/anon key, Sentry DSN |
| Domínio e SSL | Configurar domínio personalizado + SSL automático |
| Pipeline de deploy | GitHub Action com deploy automático no push para main |
| Remoção de configuração provisória | Remover wrangler.json/toml se Cloudflare não for a plataforma final |
| Sentry em produção | Configurar source maps, alertas de erro, performance tracing |
| Health check | Endpoint `/api/health` com verificação de dependências (Supabase, gateway) |

#### 4.1.3 Arquitetura de Deploy

```
Git push → main
    ↓
GitHub Actions: quality gates (lint → test → build)
    ↓
GitHub Actions: deploy (npx vercel deploy --prod)
    ↓
Vercel: SSL, CDN, Edge Functions
    ↓
Sentry: error tracking + performance monitoring
```

### 4.2 Bloco B — UX Excellence

#### 4.2.1 Princípios

- **Mobile-first:** toda página deve ser utilizável em dispositivos móveis (largura ≥ 320px).
- **Skeleton-first:** toda tela com carregamento assíncrono deve exibir skeleton imediatamente.
- **Error boundaries:** toda rota deve ter `errorComponent` e `notFoundComponent`.
- **Performance nativa:** code splitting automático (TanStack Start já faz), lazy loading de componentes pesados (charts).

#### 4.2.2 Componentes Transversais

| Componente | Descrição | Aplicação |
|-----------|-----------|-----------|
| `SkeletonCard` | Card skeleton reutilizável | Todas as páginas com listas |
| `SkeletonChart` | Skeleton para gráficos | Dashboard, Patrimônio, Rentabilidade |
| `SkeletonTable` | Skeleton para tabelas | Lançamentos, Proventos |
| `PageError` | Componente de erro de página | Rotas sem `errorComponent` |
| `PageNotFound` | Componente 404 amigável | Rotas sem `notFoundComponent` |
| `MobileNav` | Navegação inferior para mobile | Layout autenticado |
| `ResponsiveContainer` | Wrapper responsivo | Layout principal |

### 4.3 Bloco C — Intelligence Layer

#### 4.3.1 Princípios

- **Regra da Neutralidade Financeira:** o sistema nunca deve influenciar decisões com opiniões ou previsões subjetivas.
- **Dados do próprio usuário:** todos os insights são baseados exclusivamente nos dados financeiros do usuário e em métricas objetivas de mercado.
- **Camada nova, não alteração de existente:** a Intelligence Layer é uma nova camada de serviços de apresentação que consomem dados existentes, não modifica domínio, application ou infraestrutura.

#### 4.3.2 Componentes de Insight

| Componente | Descrição | Dados Consumidos |
|-----------|-----------|-----------------|
| `ProventosProjectedCard` | Proventos projetados vs. realizados no mês | Proventos históricos + posições atuais |
| `MetaProgressCard` | Progresso vs. meta financeira | Metas + patrimônio atual |
| `RebalanceamentoSuggester` | Sugestão de aporte para rebalanceamento | Alocação atual vs. desejada |
| `RentabilidadeComparativa` | Rentabilidade vs. benchmarks (IBOV, IDIV, IFIX) | Histórico de rentabilidade |
| `AlertasOportunidade` | Alertas de oportunidades com base em gaps de alocação | Configurações + posições |

#### 4.3.3 Hooks de Insight

| Hook | Descrição |
|------|-----------|
| `useInsightProventosProjected` | Projeta proventos para o mês corrente com base em posições |
| `useInsightMetaProgress` | Calcula progresso percentual em relação à meta |
| `useInsightRebalanceamento` | Sugere próximos aportes para atingir alocação desejada |

### 4.4 Bloco D — Feature Completion

#### 4.4.1 FEAT-009 — Atualizações Automáticas

**Descrição:** Sempre que possível, o sistema deve atualizar informações automaticamente, reduzindo a necessidade de manutenção manual.

**Arquitetura:**

- Criar `src/presentation/features/auto-sync/` como novo módulo
- Componente `AutoSyncIndicator` — exibe status da última sincronização
- Hook `useAutoSync` — agenda sincronização periódica de cotações e proventos
- Reutilizar `syncPendingDividends` (já existente) e `getQuotes` (já existente)
- Configuração de intervalo: a cada 15 minutos quando o usuário está ativo

**Composition Root:** Reutilizar `SincronizarDadosCommand` e `syncPendingDividends` existentes.

#### 4.4.2 FEAT-008 — Notificações Inteligentes

**Descrição:** Enviar apenas notificações relevantes, evitando excesso de informações.

**Arquitetura:**

- Reutilizar `INotificationPort` (já existente) e `IAlertRepository` (já existente)
- Novo componente `NotificationBell` na navegação superior
- Novo hook `useNotificationsQuery` — busca alertas não confirmados
- Regras de negócio no domínio (NotificationService) — exempto de alteração de domínio existente

**Observação:** Este bloco depende do Bloco E (Licensing & Access) para definir quais notificações cada plano pode receber.

### 4.5 Bloco E — Licensing & Access (BK-006)

#### 4.5.1 Princípios

- **Feature Flags por Plano (FEAT-010):** cada funcionalidade pode ser disponibilizada conforme o plano do usuário.
- **Camada de autorização reutilizável:** não criar nova lógica por funcionalidade — usar o `AuthorizationService` existente.
- **Sem alteração de domínio:** `Plan.capabilities` já existe e já é usado pelo `AuthorizationService`.

#### 4.5.2 Arquitetura Proposta

A arquitetura de Licensing & Access será uma camada **exclusivamente de Presentation** que:

1. Consome `VerificarAcessoQuery` (já existente) para determinar se o usuário tem acesso a uma funcionalidade.
2. Envolve componentes sensíveis com um `RequireCapability` wrapper.
3. Redireciona ou oculta funcionalidades não autorizadas.

**Componentes:**

| Componente | Descrição |
|-----------|-----------|
| `RequireCapability` | Wrapper que renderiza children apenas se o usuário tem a capability |
| `PlanGate` | Portão de upgrade — redireciona para `/checkout` se plano não cobre a feature |
| `useCapabilityCheck` | Hook que verifica capability via `VerificarAcessoQuery` |

**ADRs pendentes:** Definir se o licenciamento será server-side (recomendado) ou client-side.

## 6. Restrições

As Restrições R-001 a R-019 permanecem integralmente em vigor. Adicionam-se:

### R-020 — Deploy Seguro (Bloco A)

Nenhum deploy em produção pode ser realizado sem: (a) quality gates verdes, (b) Sentry configurado, (c) rollback testado.

### R-021 — Insight Neutro (Bloco C)

Nenhum insight pode conter opinião subjetiva, previsão de mercado ou recomendação de compra/venda. Insights devem ser exclusivamente baseados em dados quantitativos do próprio usuário.

### R-022 — Feature Flag por Capability (Bloco E)

Toda restrição de acesso por plano deve usar o mecanismo de `Plan.capabilities` + `AuthorizationService` existente. Nenhuma nova lógica de autorização pode ser criada.

## 12. ADRs Necessários

### ADR-013-001: Plataforma de Hospedagem (Bloco A)

| Campo | Valor |
|-------|-------|
| Contexto | Atualmente usando Cloudflare Workers provisoriamente. É necessário definir a plataforma oficial. |
| Opções | (1) Vercel — menor custo de migração, TanStack Start first-class, CI/CD integrado. (2) Cloudflare Workers — já configurado, mas limitações de Node.js API. (3) Netlify — compatível, mas ecossistema menor. |
| Decisão | **Vercel** — compatibilidade nativa com TanStack Start, zero configuração de infraestrutura, deploy por git push, plano gratuito generoso. |
| Consequências | Remover configuração Cloudflare Workers (wrangler.json). Atualizar pipeline de CI/CD. |

### ADR-013-002: Feature Flags (Bloco E)

| Campo | Valor |
|-------|-------|
| Contexto | Feature flags são necessárias para liberar funcionalidades por plano de assinatura. |
| Opções | (1) Client-side — verificação no frontend, rápida mas insegura. (2) Server-side — verificação no backend, segura mas mais lenta. (3) Híbrida — server-side para dados, client-side para UI. |
| Decisão | **Híbrida** — `AuthorizationService` server-side (já existe) + `RequireCapability` client-side (novo). O servidor nunca executa comandos não autorizados; o frontend oculta UI não autorizada para melhor UX. |
| Consequências | Nenhuma alteração no `AuthorizationService`. Novos componentes apenas na Presentation Layer. |
