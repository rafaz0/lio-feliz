## Objetivo

Materializar a arquitetura definida pela PI-007 (Presentation Layer) através da implementação incremental da camada de apresentação do Lio Feliz.

Esta Engineering Work Order transforma a especificação arquitetural aprovada em uma sequência rastreável de Slices de implementação, preservando integralmente os contratos definidos pela arquitetura.

A implementação deverá produzir uma interface de usuário completa, acessível e performática, utilizando TanStack Start + Tailwind CSS + shadcn/ui, com comunicação exclusivamente via Dispatcher (Commands/Queries/DTOs) e gerenciamento de estado via TanStack Query + React Context.

---

## Escopo

### Inclui:

- Estrutura de diretórios `src/presentation/` completa (Feature-First);
- Providers (QueryProvider, AuthProvider, DispatcherProvider, ThemeProvider, TooltipProvider);
- Layouts base (root, authenticated, unauthenticated);
- Features funcionais (dashboard, carteira, movimentações, proventos, rebalanceamento, gestão fiscal, histórico, rentabilidade, configurações, importação, sincronização, relatórios, autenticação);
- Components (UI, Feature, Layout, Shared, Composition);
- Hooks (custom, shared, dispatcher, query, mutation, toast, auth);
- Queries e Mutations via TanStack Query;
- View Models e tipos locais;
- Tratamento de erros (ApplicationError → Toast/Banner/Modal/Empty State);
- Estratégias de loading (Skeleton, Spinner, Placeholder, Suspense, Optimistic Update, Background Refetch);
- Acessibilidade (WCAG 2.1 AA, ARIA, teclado, foco, contraste);
- Testes unitários, de componente, de integração, de arquitetura e E2E;
- Configurações de lint, build e tipos estritos (TypeScript, Biome, Zod).

### Não contempla:

- Implementação concreta dos Ports da Application Layer (isso é responsabilidade da Infrastructure Layer – EWO-004);
- Persistência direta (localStorage, IndexedDB, cookies) para dados de domínio (apenas preferências de UI são permitidas);
- Acesso direto ao Supabase client ou a APIs externas (Yahoo, CoinGecko, B3) – todo acesso deve ocorrer via Dispatcher;
- Definição ou modificação de regras de negócio (cálculo de preço médio, validação de invariantes, eventos de domínio) – estas pertencem ao Domain Layer;
- Alteração de Contracts da Application Layer (Commands, Queries, DTOs, Errors, IDispatcher);
- Implementação de autenticação/autorização própria – deve consumir o Port de Auth da Infrastructure.

## Premissas Arquiteturais

- **PI-007 é a única fonte de arquitetura.**
- **A Core Foundation (EWO-001) permanece congelada.**
- **A Domain Layer (EWO-002) permanece inalterada.**
- **A Application Layer (EWO-003) permanece inalterada.**
- **A Infrastructure Layer (EWO-004) permanece inalterada.**
- **Baseline Lock obrigatório (GOV-M06):** PI-007, ER-007 e EWO-005 entram em regime de congelamento no Gate de Entrada.
- **Clean Architecture preservada:** Dependency Rule, DDD, Ports & Adapters, Dispatcher Pattern inalterados.
- **Dispatcher como único ponto de entrada:** Toda comunicação com a Application Layer ocorre exclusivamente via `IDispatcher`.
- **Feature-First:** Organização por features em `src/presentation/features/`.
- **Gerenciamento de estado oficial:** TanStack Query (Server State) + React Context + useReducer (Client State) + React Hook Form (Form State).
- **Acessibilidade como requisito arquitetural:** WCAG 2.1 AA como mínimo.
- **Testabilidade como requisito:** Arquitetura protegida por Architecture Tests (R-10).

## Estratégia de Implementação

A implementação seguirá uma abordagem incremental por Slices, onde cada Slice:

- Produz um incremento completo, compilável e testável;
- Respeita as dependências naturais entre componentes (ordem técnica, não apenas funcional);
- É completamente funcional antes da próxima Slice subsequente;
- Encerra com Build, Lint, Testes, Commit, Push e Relatório de Sincronização;
- É sujeita a auditoria do ChatGPT antes de autorizar a próxima Slice;
- Não introduz arquitetura – apenas materializa o aprovado na PI-007;
- Mantém o Baseline Lock durante toda a execução.

A ordem das Slices reflete a ordem de dependências entre componentes da Presentation Layer, garantindo que cada Slice construa sobre a anterior sem retrabalho.

## Planejamento das Slices

A seguir, a lista oficial de Slices para a EWO-005. Cada Slice detalhada posteriormente.

### Slice 1 — Foundation

- estrutura da Presentation
- Providers (Query, Auth, Dispatcher, Theme)
- Tema inicial (light/dark)
- Layout base (root, authenticated, unauthenticated)
- Configuração do TanStack Query
- Configuração do Dispatcher Provider
- Estrutura de diretórios básica

### Slice 2 — Autenticação

- Páginas de login, registro e recuperação de senha
- Guards de rota (auth required, guest only)
- Gerenciamento de sessão via AuthProvider
- Integração com o Port de Auth da Infrastructure (via Dispatcher)
- Tratamento de erros de autenticação (AUTHORIZATION_ERROR)
- Loading states e validação de formulário (Zod + RHF)

### Slice 3 — Dashboard

- Página principal `/dashboard`
- Cards de resumo (Patrimônio, Investido, Disponível, Rendimento mensal)
- Gráfico de alocação (pizza/barra)
- Integração com `usePatrimonio` (Query)
- Loading states (Skeleton >300ms, Spinner <300ms)
- Error states e Empty State
- Refresh manual e automático (window focus, interval)

### Slice 4 — Carteira (Portfolio)

- Listagem de carteiras `/portfolio`
- Detalhe da carteira `/portfolio/:id`
- Positions list e detalhe `/portfolio/:id/positions/:assetId`
- Integração com `usePatrimonio`, `usePosicaoDetalhada`
- Gráficos de alocação e evolução (reutilizando componentes de Dashboard)
- Navegação entre carteira e positions

### Slice 5 — Movimentações (Operations)

- Registro manual de operação `/portfolio/:id/operations/new`
- Importação de carteira `/portfolio/:id/operations/import`
- Formulário de operação (compra, venda, dividendo, JCP, bonificação, split, grouping, amortização, ajuste)
- Validação Zod no frontend + backend (defesa em profundidade)
- Integração com `useRegisterOperation` e `useImportPortfolioMutation`
- Toast de sucesso e tratamento de erros (ApplicationError)
- Preview de arquivo CSV (para importação)
- Limpeza de formulário após sucesso

### Slice 6 — Proventos (Dividends/JCP)

- Página `/portfolio/:id/dividends`
- Tabela de proventos com filtros (ano, ticker)
- Integração com `useProventos`
- Totais por período e acumulados
- Exportação para relatório fiscal (link para `/reports/tax`)
- Loading states (Skeleton/Spinner)
- Paginação simples (se necessário) ou virtualização (>50 linhas)

### Slice 7 — Histórico e Rentabilidade

- Histórico de patrimônio `/portfolio/:id/history`
  - Gráfico de linha/área (valor investido vs valor atual)
  - Integração com `useHistoricoPatrimonial`
- Rentabilidade por período `/portfolio/:id/performance`
  - Gráfico de barra/linha (rentabilidade mensal, anual, acumulada)
  - Integração com `useRentabilidade`
  - Comparação com benchmarks (IBOV, IDIV, IFIX)
- Shared components para gráficos (reutilizáveis entre features)

### Slice 8 — Rebalanceamento

- Página `/portfolio/:id/rebalancing`
- Alocação atual vs desejada (barras/lado a lado)
- Sugestão de aportes para rebalanceamento
- Integração com `useRebalanceamento` (Query) e `useConfigurarEstrategiaMutation` (para atualização de estratégia)
- Simulação de impacto dos aportes
- Link para `/settings/strategy` para ajuste de porcentagens

### Slice 9 — Gestão Fiscal (Imposto de Renda)

- Página `/portfolio/:id/reports/tax`
- Seleção de ano
- IRRF: posição em 31/12, proventos, ganhos de capital, prejuízos a compensar
- Integração com `useRelatorioFiscal`
- Exportação PDF/CSV via `ExportarDadosQuery` → `Download`
- Layout em seções claras (collapsible ou abas)
- Loading states

### Slice 10 — Configurações

#### 10.1 Estratégia
- Página `/settings/strategy`
- Percentuais por classe de ativo (slider ou input)
- Moeda principal (select)
- Metas financeiras (valor, data)
- Integração com `useConfigurarEstrategiaMutation` e `useConsultarProgressoMetasQuery`
- Pré-visualização da nova alocação

#### 10.2 Metas
- Página `/settings/goals`
- Lista de metas (valor alvo, data atual, progresso)
- Criação/edição/exclusão de metas
- Integração com `useConsultarProgressoMetasQuery` e `useConfigurarEstrategiaMutation`

#### 10.3 Assinatura
- Página `/settings/subscription`
- Plano atual (gratuito/premium)
- Upgrade/downgrade (via `useGerenciarAssinaturaMutation`)
- Recursos disponíveis por plano
- Renewal date e próximo pagamento

### Slice 11 — Sincronização e Importação Avançada

- Página `/sync` (execução manual ou visualização de último run)
- Botão "Sincronizar agora" (dispara `SincronizarDadosCommand`)
- Toast com resumo (importadas, rejeitadas, erros)
- Status da última sincronização (timestamp, sucesso/erro)
- Integração com `useSincronizarDadosMutation`

### Slice 12 — Relatórios

- Reutiliza páginas de `/reports/tax` e `/reports/export` (já definidas em Slices 9 e 5/6)
- Exportação de dados brutos (`ExportarDadosQuery` → `Download`)
- Layout consistente com outras páginas de relatório
- Seleção de intervalo de datas (para exportação personalizada)

### Slice 13 — Engineering Closure

- Revisão final de toda a implementação
- Testes de integração end-to-end (fluxos críticos de usuário)
- Testes de arquitetura (verificando ausencia de violações de domínio/infra, Dispatcher-only)
- Cobertura de testes >90%
- Engenharia de limpeza (remoção de código morto, ajuste de lint)
- Atualização de documentação (se necessário)
- Preparação para o Git Commit final e Engineering Closure oficial

## Plano de Testes

### Estratégia Geral de Testes

Cada Slice deve possuir cobertura de testes adequada, conforme o tipo de mudança:

- **Testes Unitários**: funções puras, hooks customizados, validações Zod, formatadores.
  - Ferramenta: Vitest + React Testing Library
  - Cobertura alvo: >90% das linhas novas/modificadas
  - Mocks: Dispatcher, TanStack Query, Auth Provider, Serviços externos (via MSW ou mocks manuais)

- **Testes de Componentes**: componentes renderizados com props e estados.
  - Ferramenta: React Testing Library + Vitest
  - Foco: interação do usuário (cliques, input, submit), rendering condicional (loading/error/empty), acesso a teclado

- **Testes de Integração**: fluxos que envolvem múltiplas camadas da Presentation Layer (ex.: formulário → Dispatcher → Query → atualização de UI).
  - Ferramenta: Vitest + MSW (Mock Service Worker) para simular respostas do Dispatcher
  - Ou: Playwright para testes de navegador completo (quando aplicável)

- **Testes de Arquitetura (Obrigatórios a partir da Slice 1)**: garantir que a Presentation Layer nunca viole a Clean Architecture.
  - Regras:
    - Nenhum import de `src/infrastructure/*`
    - Nenhum import de `src/core/domain/*`
    - Nenhum acesso direto ao Supabase client
    - Nenhuma instância direta de Application Service
    - Comunicação exclusivamente via `useDispatcher` ou hooks que o utilizam internamente
  - Ferramenta: Vitest + AST parsing (ts-morph ou similar) ou ESLint plugin customizado
  - Execução: em cada CI/push e localmente antes do commit

- **Testes E2E (quando aplicável)**: fluxos críticos de usuário que envolvem múltiplas features e persistência simulada.
  - Ferramenta: Playwright
  - Cenários: login → onboarding → importação de carteira → registro de operação → visualização do dashboard → exportação de relatório → logout
  - Ambiente: usando o modo DEV (`DEV_MODE=true`) com armazenamento em memória

### Execução dos Testes

- Antes de cada commit: `npm run test` (unit + component) e `npm run test:arch` (arquitetura)
- Antes de cada push: `npm run test:e2e` (se houver cenários definidos)
- No CI: pipeline completo (build, lint, todos os testes)

## Rollback (Procedimento de Reversão)

Cada Slice deve ser reversível via Git, já que não há alterações de banco de dados ou estado externo persistente (apenas preferências de UI em localStorage, que são triviais de limpar ou ignorar).

- **Procedimento**: `git revert <commit-da-slice>` ou `git reset --hard <commit-anterior>` (se ainda não houver push)
- **Após o push**: criar um novo commit que reverte as alterações indesejadas, seguindo a mesma estrutura de commit.
- **State do cliente**: caso tenham sido armazenadas preferências de UI (tema, sidebar aberta), estas não afetam a consistência do domínio e podem ser mantidas ou resetadas conforme convenção da aplicação (ex.: sempre iniciar com tema claro e sidebar fechada).
