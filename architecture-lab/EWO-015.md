## Objetivo

Materializar o Bloco C da PI-012 — Comercialização — substituindo o billing simulado por um gateway de pagamento real e criando um fluxo de checkout integrado.

Os pilares desta EWO são:

1. **IPaymentGateway** — Interface de gateway substituindo `BillingSimulator`, sem alterar domínio
2. **Billing Real** — Cobrança mensal automática com retry e webhook
3. **Checkout Integrado** — Fluxo combinando onboarding + assinatura + pagamento

Nenhum Core Domain, entidade, VO ou regra de negócio é alterado.

---

## Escopo

### Inclui:

- Interface `IPaymentGateway` com métodos charge, cancel, refund (ADR-012-001)
- Adapter Stripe (concreto) implementando `IPaymentGateway`
- Substituição do `BillingSimulator` pelo `IPaymentGateway` no `AssinarPlanoService`
- Job agendado de cobrança mensal
- Retry com backoff (D+0, D+3, D+7)
- Webhook de confirmação de pagamento
- `CheckoutOrchestrator` (Application Layer) compondo `AssinarPlanoService` + `OnboardingFlow` + `IPaymentGateway` (ADR-012-002)
- Componente `CheckoutForm` (Presentation)
- Atualização documental

### Não contempla:

- Modificação de Core Domain (Plan, Subscription, BillingCycle inalterados)
- Modificação de Application Services existentes (AssinarPlanoService mantido)
- Criação de novos commands ou queries de domínio
- Criação de novos ports de infraestrutura
- Configuração de secrets da Cloudflare

---

## Princípios Obrigatórios

1. **Nenhuma alteração de domínio** — Plan, Subscription, BillingCycle, entities, VOs, domain services permanecem inalterados
2. **Adapter substituível (R-019)** — `IPaymentGateway` como interface; `BillingSimulator` é substituído, não alterado
3. **Orquestração, não novo comando (ADR-012-002)** — Checkout compõe serviços existentes, não cria novos
4. **Frozen Baselines preservadas**
5. **Feature-First** para CheckoutForm

---

## Estratégia de Implementação

A ordem segue a dependência entre os componentes:

1. `IPaymentGateway` (interface necessária para billing e checkout)
2. Billing real (job + retry + webhook)
3. Checkout integrado (depende de gateway e billing)

#### Interface

Criar `src/application/gateways/payment-gateway.ts`:

```typescript
export interface IPaymentGateway {
  charge(subscriptionId: string, amount: number): Promise<PaymentResult>;
  cancel(subscriptionId: string): Promise<void>;
  refund(chargeId: string): Promise<void>;
}

export interface PaymentResult {
  success: boolean;
  transactionId: string;
  status: "PAID" | "FAILED" | "REFUNDED";
  error?: string;
}
```

#### Adapter Stripe

Criar `src/infrastructure/gateways/stripe-payment-gateway.ts`:

- Implementa `IPaymentGateway` usando Stripe SDK
- `charge` → `stripe.paymentIntents.create`
- `cancel` → `stripe.paymentIntents.cancel`
- `refund` → `stripe.refunds.create`
- Construtor recebe `stripeSecretKey: string`

- Se gateway presente, chama `charge` após criar subscription
- Se gateway ausente, usa `BillingSimulator` como fallback (retrocompatibilidade)

### Slice 2 — Billing Real

| Campo | Valor |
|-------|-------|
| Número | 2 |
| Objetivo | Implementar cobrança mensal automática, retry e webhook |
| Status | PENDENTE |

#### Job de Cobrança Mensal

Criar `src/application/services/billing-job-service.ts`:

- Executa `PaymentGateway.charge` para todas as subscriptions ACTIVE no início do mês
- Cria `BillingCycle` para cada cobrança
- Atualiza status do ciclo (PAID / FAILED)

#### Retry com Backoff

- 3 tentativas: D+0 (imediata), D+3, D+7
- Se todas falharem, subscription é cancelada (CANCELLED)
- Cada tentativa gera um `BillingCycle` com status FAILED

#### Webhook

Criar server function `src/routes/api.stripe-webhook.ts`:

- Endpoint POST `/api/stripe-webhook`
- Recebe confirmação do Stripe (event `payment_intent.succeeded`)
- Atualiza `BillingCycle.status = "PAID"`
- Retorna 200 OK

#### Testes

- Testes unitários do BillingJobService
- Testes de retry com FakePaymentGateway

#### Critérios de Aceite

- [ ] BillingJobService executa cobrança para subscriptions ativas
- [ ] Retry com 3 tentativas implementado
- [ ] Subscription cancelada após falha em todas as tentativas
- [ ] Webhook endpoint criado
- [ ] BillingCycle com status PAID/FAILED/REFUNDED preservado

#### CheckoutOrchestrator

Criar `src/application/services/checkout-orchestrator.ts`:

- Não é um comando — é um orquestrador que compõe serviços existentes
- Fluxo:
  1. `OnboardingFlow.completeStep(progress, step)` — avança onboarding
  2. `AssinarPlanoService.Execute(command)` — cria subscription
  3. `PaymentGateway.charge(subscriptionId, amount)` — cobra

```typescript
export interface CheckoutInput {
  userId: string;
  planId: string;
  answers?: import("@/core/domain/investor-profile").Answer[];
  paymentMethodId: string;
}

export class CheckoutOrchestrator {
  constructor(
    private readonly onboardingFlow: OnboardingFlow,
    private readonly assinarPlanoService: AssinarPlanoService,
    private readonly paymentGateway: IPaymentGateway,
  ) {}

  async execute(input: CheckoutInput): Promise<CheckoutResult> {
    // 1. Subscription
    const subscription = await this.assinarPlanoService.Execute({
      type: "AssinarPlanoCommand",
      planId: input.planId,
      userId: input.userId,
    });

    // 2. Cobrança
    const payment = await this.paymentGateway.charge(subscription.id, 0);

    return { subscription, payment };
  }
}
```
