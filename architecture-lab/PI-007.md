## 1. Objetivo

Esta Product Interpretation (PI-007) estabelece a arquitetura oficial da Presentation Layer do Lio Feliz.

Seu objetivo é definir como a camada de apresentação consome os contratos públicos da Application Layer (Commands, Queries, DTOs, Ports expostos) para construir a experiência do usuário, mantendo total independência tecnológica em relação ao domínio e à infraestrutura.

A Presentation Layer é a camada que materializa os 14 Casos de Uso (UC-001 a UC-014) e os 34 Requisitos Funcionais (FR-001 a FR-034) em interfaces interativas, acessíveis e performáticas.

Esta PI estabelece:

- a estrutura de diretórios `src/presentation/` e sua organização por features;
- os princípios arquiteturais que garantem desacoplamento do domínio e da infraestrutura;
- a estratégia de comunicação com a Application Layer via Dispatcher (Commands/Queries);
- a organização funcional em módulos alinhados à Especificação Funcional (21_FUNCTIONAL_SPECIFICATION.md);
- os fluxos de usuário principais derivados da Jornada Completa do Usuário;
- a estratégia oficial de gerenciamento de estado;
- convenções de nomenclatura, organização e reutilização de componentes;
- restrições explícitas do que a Presentation Layer **não** pode fazer.

## 2. Princípios Arquiteturais

A Presentation Layer deverá respeitar permanentemente os seguintes princípios.

### PA-007-001 — Consumo Exclusivo de Contratos Públicos

A Presentation Layer **apenas** consome:
- Commands e Queries definidos na Application Layer (PI-005);
- DTOs de resposta definidos na Application Layer;
- Application Errors definidos na Application Layer;
- Ports de autenticação/autorização expostos pela Infrastructure.

Ela **nunca** acessa diretamente:
- Entidades do Domínio (PI-004);
- Repositórios, Gateways, Unit of Work ou quaisquer adapters de Infrastructure;
- Banco de dados, APIs externas, Supabase client diretamente (exceto para autenticação via Port);
- Eventos de domínio (Domain Events) — estes são consumidos exclusivamente pela Application Layer.

### PA-007-002 — Clean Architecture — Dependency Rule

A direção de dependências é obrigatoriamente:

```
Presentation Layer
    ↓ (depende)
Application Layer
    ↓ (depende)
Domain Layer
    ↓ (depende)
Core Foundation (C-001 + C-002)

Infrastructure Layer → implementa Ports da Application Layer
```

A Presentation Layer **conhece** a Application Layer. A Application Layer **não conhece** a Presentation Layer.

### PA-007-003 — Ports & Adapters na Fronteira

A comunicação entre Presentation e Application ocorre exclusivamente através do **Dispatcher** (definido na PI-005, Seção 5.3).

- Commands são enviados ao Dispatcher → Application Service → Domínio → UoW → DTO
- Queries são enviadas ao Dispatcher → Application Service → Projeção/Leitura → DTO

Nenhum componente da Presentation Layer instancia diretamente um Application Service.

### PA-007-004 — DDD — Camada de Aplicação como Anti-Corruption

A Presentation Layer atua como **Anti-Corruption Layer** entre o mundo externo (usuário, navegador, dispositivos) e a Application Layer.

Ela é responsável por:
- Transformar entrada do usuário (formulários, parâmetros de URL, eventos de UI) em Commands/Queries tipados;
- Transformar DTOs de saída em View Models adequados à renderização;
- Traduzir Application Errors em feedback amigável ao usuário (toasts, mensagens inline, páginas de erro);
- Gerenciar estados de UI (loading, empty, error, success) que não existem na Application Layer.

### PA-007-005 — Baseline Lock (GOV-M06)

As quatro camadas-base estão **congeladas**:
- Core Foundation (C-001 + C-002) — GOV-006
- Domain Layer (EWO-002) — PI-004
- Application Layer (EWO-003) — PI-005
- Infrastructure Layer (EWO-004) — PI-006

A PI-007 **não pode** modificar, estender ou contornar contratos, invariantes, decisões arquitetuais ou APIs públicas das camadas congeladas. Qualquer necessidade de alteração deve seguir o fluxo oficial: nova PI → ER → EWO → Baseline Lock.

### PA-007-006 — Separação de Responsabilidades por Feature

A organização do código segue **Feature-First** (não Layer-First). Cada feature funcional agrupa seus próprios componentes, hooks, tipos e testes, reduzindo acoplamento entre features.

### PA-007-007 — Acessibilidade como Requisito Arquitetural

Acessibilidade (WCAG 2.1 AA) não é opcional nem pós-processamento. Componentes base (shadcn/ui) já fornecem base acessível; componentes compostos devem preservar semântica ARIA, navegação por teclado, foco visível e contraste.

## 4. Comunicação com a Application Layer

### 4.1 Dispatcher como Único Ponto de Entrada

A PI-005 (Seção 5.3) define o `IDispatcher` como contrato único. A Presentation Layer obtém o dispatcher via React Context (`DispatcherProvider`) e o usa através de hooks tipados.

```typescript
// shared/hooks/use-dispatcher.ts
export function useDispatcher() {
  const context = useContext(DispatcherContext);
  if (!context) throw new Error('DispatcherProvider não encontrado');
  return context;
}
```

### 4.1x — Estado do Dispatcher (Contexto React)

O dispatcher é exposto através de um Provider Context para que hooks possam acessá-lo onde quer que sejam necessários.

```typescript
// shared/context/DispatcherContext.tsx
import { createContext, useContext } from 'react';
import type { IDispatcher } from 'core/application/dispatcher';

type DispatcherContextType = IDispatcher | null;

export const DispatcherContext = createContext<DispatcherContextType>(null);

export function useDispatcher(): IDispatcher {
  const context = useContext(DispatcherContext);
  if (!context) {
    throw new Error('useDispatcher deve ser usado dentro de um DispatcherProvider');
  }
  return context;
}
```

```typescript
// shared/providers/DispatcherProvider.tsx
import { DispatcherContext } from '../context/DispatcherContext';
import type { IDispatcher } from 'core/application/dispatcher';

interface DispatcherProviderProps {
  dispatcher: IDispatcher;
  children: React.ReactNode;
}

export function DispatcherProvider({ dispatcher, children }: DispatcherProviderProps) {
  return (
    <DispatcherContext.Provider value={dispatcher}>
      {children}
    </DispatcherContext.Provider>
  );
}
```

**R-01 — Matriz Oficial de Gerenciamento de Estado**

A Presentation Layer utiliza **duas camadas primárias de estado** com responsabilidades claramente definidas:

| Tipo de Estado | Tecnologia | Escopo | Responsabilidades |
|----------------|------------|--------|------------------|
| **Server State** | TanStack Query | Global (QueryClient) | **Dados síncronos** provenientes da Application Layer:
- Dados derivados de Commands (`OperacaoRegistradaDto`, `ImportacaoRealizadaDto`, etc.)
- Dados derivados de Queries (`PatrimonioDto`, `PosicaoDetalhadaDto`, etc.)
- Listas e conjuntos de dados (Portfolio[], Positions[], etc.)
- **Regras:** Cache, invalidação, refetch em background, gerenciamento de paginação |
| **Client State** | React Context + useReducer | Feature / Componente | **Estado de UI** local:
- Estados de loading/empty/error por view
- Abrir/fechar modal, sidebar, drawer
- Theme/dark-mode (shadcn/ui)
- Filtros de tabela, busca local, ordenação
- Inputs de formulário (React Hook Form gerencia estado do formulário, mas UI controla visibilidade, help text, estado de campo) |
| **Form State** | React Hook Form + Zod | Componente | Schemas de validação (Zod), serialização de payload para dispatcher |
| **Auth State** | Supabase Auth + React Context | Global | Autenticação, permissões, refresh token, ciclo de sessão |

**Princípios de design:**

1. **Camada única de truth:** O estado do servidor é a única fonte de truth para dados da Application Layer.
2. **State local efêmero:** Estado de UI existe exclusivamente no Client State; nunca duplica dados do servidor.
3. **Regras de invalidação:** Toda mutation (Command) **deve** invalidar queries relacionadas via queryKey canônico.
4. **Gerenciamento de loading states:** Cada query/mutation deve expor states `idle`, `loading`, `success`, `error` através de uma interface `BaseViewModel<T>` padronizada.
5. **Error handling:** Erros Application são mapeados para estados de erro de UI padronizados; view models exibem erros específicos por feature.

**Organização do Estado por Feature:**

Cada feature pode definir um **contexto de estado local** se precisar de aggregations complexas (ex.: estado de modal, filtros avançados). Contexto local não deve substituir state global quando gerenciamento centralizado for mais apropriado.

**Recomendação de implementação:**

```typescript
// shared/hooks/useQueryState.ts
export function useQueryState<T>(queryKey: string[], enabled: boolean = true) {
  return useQuery<T>({
    queryKey,
    enabled,
    staleTime: 30_000,
    retry: (failureCount, error) => {
      if (failureCount >= 3) return false;
      if (error instanceof ApplicationError && error.type === 'AUTHORIZATION_ERROR') return false;
      return true;
    },
  });
}
```

```typescript
// shared/hooks/useMutation.ts
export function useMutation<TData, TError, TVariables>(options: { mutationFn: (variables: TVariables) => Promise<TData>; }) {
  return useMutation<TData, TError, TVariables>({
    ...options,
    onSuccess: (data) => {
      // Lógica de invalidação baseada em tipo de retorno
    },
  });
}
```

**Melhoria futura potencial:** Implementar middleware de estado com hooks de subscription para reagir a mudanças de server state em toda a aplicação (ex.: notificação de reposição após imports de carteira bem-sucedidos).

**R-02 — Classificação Oficial dos Componentes**

Documentar oficialmente as categorias de componentes da Presentation Layer.

**Sugerido:**

- **UI Components** — Componentes base reutilizáveis, estilizados com Tailwind (Buttons, Inputs, Cards, Tables, Modals, etc.)
- **Feature Components** — Componentes compostos específicos de features que combinam UI Components com hooks e view models
- **Layout Components** — Layouts raiz, sidebars, headers, footers, wrapper de page
- **Shared Components** — Componentes compartilhados entre features (ex.: DataTable, FilterPanel, SearchBox)
- **Providers** — Wrappers context (DispatcherProvider, QueryProvider, AuthProvider, ThemeProvider)

**Explicação da responsabilidade de cada categoria:**

1. **UI Components** — Componentes cosméticos/semântica de apresentação, stateless onde possível
2. **Feature Components** — Contêm lógica de apresentação da feature, hooks customizados, view models
3. **Layout Components** — Define a estrutura da página/layout em si
4. **Shared Components** — Reutilizáveis em múltiplas features, abstraem padrões de UI comuns
5. **Providers** — Gerenciam contextos globais, clients externos, estado derivado de dados da Application Layer

**R-04 — Estratégia TanStack Start**

**Documentar explicitamente quando utilizar:**

1. **Server Components** (se aplicável)
2. **Client Components**
3. **Server Functions**
4. **SSR**
5. **CSR**

**Implementação:**

A Presentation Layer adota uma **estratégia híbrida Server-First**:

- **Pages** que carregam dados léxicos são implementadas como **Server Components** quando possível, aproveitando SSR para SEO e performance inicial.
- **Componentes** interativos que precisam de estado cliente são **Client Components**.
- **Server Functions** são utilizadas apenas quando necessário para lógica não-UI (ex.: busca de dados que não têm representação UI direta).
- **SSR** é aplicada a rotas de página que possuem dados léxicos estáveis.
- **CSR** é a estratégia padrão para componentes interativos.

**Decisão de design baseada em caso de uso:**

```typescript
// app/routes/dashboard.route.tsx (Server Component)
export default async function DashboardPage() {
  const { data: patrimonio, isLoading } = useQuery({
    queryKey: ['patrimonio'],
    queryFn: async () => await fetchPatrimonio(),
  });

  if (isLoading) return <LoadingSkeleton />;

  return <DashboardView data={patrimonio} />;
}
```

**Política de fallback de estratégia:**

Se `routing.ts` estiver configurado para `config.server.deep():
- Rotas que chamam Server Functions → SSR + Client hydration
- Rotas sem Server Functions → CSR

Se `config.server.isolation` estiver definido como `auto`:
- Rotas com apenas dados léxicos → SSR
- Rotas com interatividade esperada → CSR

**Regras de validação de consistência:**

1. Componentes compartilhados nunca devem fazer referência a dados léxicos sem serem Server Components.
2. Client Components nunca devem instanciar `fetch` de servidor diretamente; devem usar `tanstack-query`.
3. Server Functions devem ser tipadas como `export const GET_PATRIMONIO = server$('/api/patrimonio')` (syntax de TanStack Start).

| Tipo de Estado | Tecnologia | Escopo | Responsabilidades |
|----------------|------------|--------|------------------| |
| **Server State** | TanStack Query | Global (QueryClient) | **Dados síncronos** provenientes da Application Layer:
- Dados derivados de Commands (`OperacaoRegistradaDto`, `ImportacaoRealizadaDto`, etc.)
- Dados derivados de Queries (`PatrimonioDto`, `PosicaoDetalhadaDto`, etc.)
- Listas e conjuntos de dados (Portfolio[], Positions[], etc.)
- **Regras:** Cache, invalidação, refetch em background, gerenciamento de paginação |
| **Client State** | React Context + useReducer | Feature / Componente | **Estado de UI** local:
- Estados de loading/empty/error por view
- Abrir/fechar modal, sidebar, drawer
- Theme/dark-mode (shadcn/ui)
- Filtros de tabela, busca local, ordenação
- Inputs de formulário (React Hook Form gerencia estado do formulário, mas UI controla visibilidade, help text, estado de campo) |
| **Form State** | React Hook Form + Zod | Componente | Schemas de validação (Zod), serialização de payload para dispatcher |
| **Auth State** | Supabase Auth + React Context | Global | Autenticação, permissões, refresh token,命理状态不稳定时重试 |

**Princípios de design:**

1. **Camada única de truth:** O estado do servidor é a única fonte de truth para dados da Application Layer.
2. **State local efêmero:** Estado de UI existe exclusivamente no Client State; nunca duplica dados do servidor.
3. **Regras de invalidação:** Toda mutation (Command) **deve** invalidar queries relacionadas via queryKey canônico.
4. **Gerenciamento de loading states:** Cada query/mutation deve expor states `idle`, `loading`, `success`, `error` através de uma interface `BaseViewModel<T>` padronizada.
5. **Error handling:** Erros Application são mapeados para estados de erro de UI padronizados; view models exibem erros específicos por feature.

**Organização do Estado por Feature:**

Cada feature pode definir um **contexto de estado local** se precisar de aggregations complexas (ex.: estado de modal, filtros avançados). Contexto local não deve substituir state global quando gerenciamento centralizado for mais apropriado.

**Recomendação de implementação:**

Commands alteram estado. Fluxo obrigatório:

```typescript
// features/portfolio/mutations/register-operation.mutation.ts
import { useMutation } from '@tanstack/react-query';
import { useDispatcher } from '@/shared/hooks/use-dispatcher';
import type { RegistrarOperacaoCommand } from 'core/application/commands';
import type { OperacaoRegistradaDto } from 'core/application/dtos';

export function useRegisterOperation() {
  const dispatcher = useDispatcher();

  return useMutation<OperacaoRegistradaDto, ApplicationError, RegistrarOperacaoCommand>({
    mutationFn: async (command) => {
      const result = await dispatcher.dispatch(command);
      if (result instanceof ApplicationError) throw result;
      return result;
    },
    onSuccess: (data) => {
      // Invalidate queries relacionadas
      queryClient.invalidateQueries({ queryKey: ['portfolio', data.portfolioId] });
      queryClient.invalidateQueries({ queryKey: ['positions', data.portfolioId] });
      toast.success('Operação registrada com sucesso');
    },
    onError: (error) => {
      toast.error(formatApplicationError(error));
    },
  });
}
```

**Regras:**
- Commands **sempre** usam `useMutation` (TanStack Query);
- Invalidação de queries relacionada **obrigatória** no `onSuccess`;
- Tratamento de `ApplicationError` padronizado via `formatApplicationError`;
- Feedback ao usuário via Toast (sucesso) ou inline/form error (validação).

### 4.3 Consumo de Queries

Queries consultam estado. Fluxo obrigatório:

```typescript
// features/portfolio/queries/portfolio.query.ts
import { useQuery } from '@tanstack/react-query';
import { useDispatcher } from '@/shared/hooks/use-dispatcher';
import type { ObterPatrimonioQuery } from 'core/application/queries';
import type { PatrimonioDto } from 'core/application/dtos';

export function usePatrimonio(portfolioId: string) {
  const dispatcher = useDispatcher();

  return useQuery<PatrimonioDto, ApplicationError>({
    queryKey: ['patrimonio', portfolioId],
    queryFn: async () => {
      const query: ObterPatrimonioQuery = { portfolioId };
      const result = await dispatcher.dispatch(query);
      if (result instanceof ApplicationError) throw result;
      return result;
    },
    enabled: !!portfolioId,
    staleTime: 30_000, // 30s cache
  });
}
```

**Regras:**
- Queries **sempre** usam `useQuery` (TanStack Query);
- `staleTime` mínimo de 30s para evitar refetch excessivo;
- `queryKey` padronizado: `['feature', 'resource', id?]`;
- `enabled` condicional quando depende de parâmetros.

### 4.4 Tratamento de Erros

```typescript
// shared/utils/format-application-error.ts
export function formatApplicationError(error: ApplicationError): string {
  switch (error.type) {
    case 'VALIDATION_ERROR':
      return error.details?.map(d => `${d.field}: ${d.message}`).join(', ') ?? 'Dados inválidos';
    case 'NOT_FOUND_ERROR':
      return 'Recurso não encontrado';
    case 'AUTHORIZATION_ERROR':
      return 'Você não tem permissão para esta ação';
    case 'CONCURRENCY_ERROR':
      return 'Os dados foram alterados por outro processo. Recarregue e tente novamente.';
    default:
      return 'Erro inesperado. Tente novamente.';
  }
}
```

Erros de validação (`VALIDATION_ERROR`) são exibidos **inline** no formulário (React Hook Form `setError`).
Erros de negócio/autorização/not-found são exibidos via **Toast** ou **Página de Erro** dedicada.

### 4.5 Loading, Empty, Error States

Cada hook de query/mutation expõe estados padronizados:

```typescript
// shared/types/view-model.ts
export type LoadingState = 'idle' | 'loading' | 'success' | 'error';

export interface BaseViewModel<T> {
  data: T | null;
  state: LoadingState;
  error: ApplicationError | null;
  isLoading: boolean;
  isError: boolean;
  isSuccess: boolean;
}
```

Componentes **devem** renderizar:
- `Skeleton` (shadcn/ui) durante `loading` inicial;
- `EmptyState` componente quando `data === null` && `!isLoading`;
- `ErrorState` componente quando `isError` com botão "Tentar novamente";
- Conteúdo principal quando `isSuccess`.

## R-06 — Fonte Oficial da Verdade (State Management)

### Estado Oficial

A única fonte de verdade do sistema continua sendo a **Application Layer**.

A Presentation Layer apenas mantém uma representação temporária destes dados através do TanStack Query.

**Hierarquia de Estado Oficial:**

| Camada | Responsabilidade |
|--------|------------------|
| **Domain** | Regras de negócio e invariantes |
| **Application** | **Estado oficial** — coordenação de casos de uso, transações, DTOs |
| **Infrastructure** | Persistência e adaptação de tecnologia |
| **Presentation** | Representação de UI — cache temporário via TanStack Query |

**Regra:** O cache do TanStack Query **não** é considerado estado oficial do sistema. É uma otimização de performance para evitar refetch desnecessário. A Application Layer permanece como a única autoridade sobre o estado do sistema.

## R-09 — Estratégia Oficial TanStack Start

### Terminologia Oficial

Substituir terminologia ambígua pela oficial:

| Termo Evitar | Termo Oficial |
|--------------|---------------|
| "Server Components" | **Server Rendering (SSR)** |
| — | **Client Components** |
| — | **Server Functions** |
| — | **Hydration** |
| — | **Streaming** (quando aplicável) |

### Estratégia Documentada

A Presentation Layer adota uma **estratégia híbrida Server-First**:

- **Páginas** que carregam dados léxicos estáveis utilizam **SSR** para performance inicial e SEO
- **Componentes** interativos que precisam de estado cliente permanecem **Client Components**
- **Server Functions** são usadas apenas para lógica não-UI (ex.: chamadas de API que não têm representação UI direta)
- **UI permanece desacoplada** da Infrastructure

**Exemplo — Server Rendering (SSR):**

```tsx
// app/routes/dashboard.route.tsx
export default async function DashboardPage() {
  const { data: patrimonio, isLoading } = useQuery({
    queryKey: ['patrimonio'],
    queryFn: async () => await fetchPatrimonio(),
  });

  if (isLoading) return <LoadingSkeleton />;

  return <DashboardView data={patrimonio} />;
}
```

**Exemplo — Client Component (CSR):**

```tsx
// features/dashboard/components/DashboardView.tsx
export default function DashboardView({ data }: { data: PatrimonioDto }) {
  const [localFilter, setLocalFilter] = useState<string>('');
  // Estado de UI local, sem persistência no servidor
  return (
    <div>
      <FilterBar onFilterChange={setLocalFilter} />
      <SummaryCards data={data} />
    </div>
  );
}
```

**Regras de Validação:**
1. Componentes compartilhados **nunca** referenciam dados léxicos (sempre Client Components)
2. Client Components **nunca** instanciam `fetch` de servidor diretamente; usam `tanstack-query`
3. Server Components são sanitizados (tipagem estrita, tipo de retorno bem definido)
4. CSR components são encapsulados com Suspense para transições suaves

## R-11 — Performance Guidelines

Seção normativa com diretrizes oficiais de performance.

| Diretriz | Descrição |
|----------|-----------|
| **Lazy Loading obrigatório** | Features grandes carregadas sob demanda via `React.lazy()` + `Suspense` |
| **Code Splitting por rota** | Cada rota principal tem seu próprio chunk |
| **Prefetch inteligente** | TanStack Query `prefetchQuery` em hover/links para navegação instantânea |
| **Virtualização** | Obrigatório para listas extensas (>50 itens) via `@tanstack/react-virtual` |
| **Memoização** | Apenas quando necessária (`React.memo`, `useMemo`, `useCallback` com dependências estáveis) |
| **Suspense** | Usado quando fizer sentido para transições de loading |
| **Skeletons** | Preferíveis a spinners em carregamentos perceptíveis (>300ms) |

**Regra:** Memoização **não** é obrigatória por padrão — apenas quando perfilamento mostrar benefício real.

## R-12 — Error Handling Strategy

### Fluxo Oficial

```
Dispatcher
    ↓
Result<T, ApplicationError>
    ↓
Presentation
    ↓
UX (Toast / Banner / Modal / Retry / Empty State / Error Boundary)
```

### Tratamento Padronizado

| Tipo de Erro | UX | Componente |
|--------------|-----|------------|
| `VALIDATION_ERROR` | Inline no formulário | `FormField` com `setError` (React Hook Form) |
| `NOT_FOUND_ERROR` | Empty State com CTA | `EmptyState` component |
| `AUTHORIZATION_ERROR` | Banner + redirect | `AuthBanner` + navegação |
| `CONCURRENCY_ERROR` | Toast + botão "Recarregar" | `Toast` + `RetryButton` |
| Erro inesperado | Error Boundary | `ErrorBoundary` component |

### Regra Fundamental

A Presentation **nunca** interpreta exceções diretamente da Infrastructure.

Sempre consome `Result<T, ApplicationError>` proveniente da Application Layer.

```typescript
// ✅ Correto — via Result do Dispatcher
const result = await dispatcher.dispatch(command);
if (result instanceof ApplicationError) {
  throw result; // Deixa o useMutation tratar
}

// ❌ Incorreto — try/catch direto em serviço de infra
try {
  await supabase.from('portfolio').insert(...);
} catch (e) { ... }
```

## R-13 — Loading Strategy

Diretrizes oficiais para estados de carregamento — garantem consistência visual entre todas as features.

| Estado | Quando Usar | Componente |
|--------|-------------|------------|
| **Skeleton** | Carregamento inicial de página/dados (>300ms) | `Skeleton` (shadcn/ui) |
| **Spinner** | Ações rápidas do usuário (<300ms) | `Spinner` / `Loader` |
| **Placeholder** | Layout conhecido, dados pendentes | `Card` com `Skeleton` interno |
| **Suspense** | Transições entre rotas/estados | `<Suspense fallback={<Skeleton />}>` |
| **Optimistic Update** | Mutations de alta frequência (like, toggle) | `onMutate` + `onError` rollback no `useMutation` |
| **Background Refetch** | Dados que podem estar stale | `staleTime` + `refetchOnWindowFocus` |

**Objetivo:** Garantir consistência visual entre todas as features — nenhuma feature deve inventar seu próprio padrão de loading.
