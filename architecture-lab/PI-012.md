## 1. Objetivo

Esta Product Interpretation (PI-012) estabelece a arquitetura para a conclusão da plataforma Lio Feliz, preparando-a para produção e comercialização.

Seus objetivos são:

- finalizar a Presentation Layer dos 4 módulos pendentes (Backtests 14, Alertas 15, Educação 17, Exportação Avançada 18);
- definir a arquitetura de engenharia para produção: CI/CD, testes E2E, performance e observabilidade;
- definir a arquitetura de comercialização: gateway de pagamento, billing real e checkout integrado.

Toda evolução respeita as Frozen Baselines (PI-001 a PI-007) e as PIs anteriores (PI-008 a PI-011). Nenhum port existente é modificado. Nenhuma regra de negócio é alterada. Nenhum Core Domain é tocado.

### 4.2 Bloco B — Production Readiness (Arquitetura)

Este bloco define a arquitetura de engenharia, não implementa código.

#### 4.2.1 CI/CD Pipeline

| Estágio | Ferramenta | Ação |
|---------|-----------|------|
| Lint | ESLint + Prettier | `npm run lint` |
| Type Check | TypeScript | `npx tsc --noEmit` |
| Testes Unitários | Vitest | `npx vitest run` |
| Testes E2E | Playwright | `npx playwright test` |
| Build | Vite (TanStack Start) | `npm run build` |
| Deploy | Wrangler (CLoudflare Workers) | `npx wrangler deploy` |

**Arquitetura do Pipeline (GitHub Actions):**

```yaml
name: ci/cd
on: [push, pull_request]
jobs:
  quality:
    runs-on: ubuntu-latest
    steps: [lint, typecheck, test, build]
  e2e:
    needs: [quality]
    runs-on: ubuntu-latest
    steps: [e2e test]
  deploy:
    needs: [e2e]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps: [deploy to cloudflare]
```

#### 4.2.2 Testes E2E (Playwright)

| Suite | Escopo | Fluxos Críticos |
|-------|--------|-----------------|
| Auth | Login, registro, logout | Fluxo completo de autenticação |
| Carteira | CRUD operações, consolidação | Lançamento, edição, exclusão |
| Proventos | Sincronização, visualização | Sincronizar, confirmar alerta |
| IRPF | Cálculo, exportação | Apuração mensal, declaração anual |
| Cobertura | Mínimo 3 fluxos por módulo principal | ~30 testes E2E |

**Arquitetura:** Playwright + Test Fixtures + Supabase test database + InMemory fallback.

#### 4.2.3 Performance

| Técnica | Aplicação | Impacto Esperado |
|---------|-----------|-----------------|
| Code Splitting | Lazy loading de features (import()) | Redução de 40% no bundle inicial |
| TanStack Query Cache | Stale-while-revalidate + persistência | Redução de 60% em requisições |
| Imagem/Asset Otimização | Vite built-in + Cloudflare Images | Redução de 80% em assets |
| Server Components | TanStack Start SSR + streaming | Redução de TTFB em 50% |

| Camada | Ferramenta | Métricas |

### 4.3 Bloco C — Comercialização (Arquitetura)

#### 4.3.1 Gateway de Pagamento

O `BillingSimulator` (módulo 19) será substituído por integração com gateway real, preservando a interface do domínio.

| Componente Atual | Componente Futuro | Contrato |
|-----------------|-------------------|----------|
| `BillingSimulator.simulateBilling(subscription)` | `PaymentGateway.charge(subscription)` | Mesma entrada/saída |
| `BillingCycle.status = "PAID"` | `BillingCycle.status = "PAID" | "FAILED"` | Adiciona FAILED |
| `BillingSimulator.cancelBilling(subscription)` | `PaymentGateway.cancel(subscription)` | Mesma interface |

**Interface do Gateway:**

```typescript
interface IPaymentGateway {
  charge(subscriptionId: string, amount: number): Promise<PaymentResult>;
  cancel(subscriptionId: string): Promise<void>;
  refund(chargeId: string): Promise<void>;
}

interface PaymentResult {
  success: boolean;
  transactionId: string;
  status: "PAID" | "FAILED" | "REFUNDED";
  error?: string;
}
```

> **Decisão O1:** O `BillingSimulator` é substituído por um adapter que implementa `IPaymentGateway`. Nenhuma entidade de domínio é alterada. O `AssinarPlanoService` continua consumindo a mesma interface.

#### 4.3.2 Billing Real

| Funcionalidade | Descrição |
|---------------|-----------|
| Cobrança mensal automática | Job agendado que executa `PaymentGateway.charge` para todas as subscriptions ativas no início do mês |
| Retry com backoff | 3 tentativas (D+0, D+3, D+7) antes de cancelar |
| Webhook de confirmação | Endpoint para receber confirmação do gateway |
| Ciclo de faturamento | BillingCycle com status PAID/FAILED/REFUNDED |

#### 4.3.3 Checkout Integrado

Fluxo de checkout combinando onboarding + assinatura:

```
1. Usuário completa Onboarding (mód. 21)
2. Seleciona plano (mód. 19)
3. Insere dados de pagamento (novo componente CheckoutForm)
4. AssinarPlanoService + PaymentGateway.charge
5. Subscription ativada
6. Redirecionamento para dashboard personalizado
```

> **Decisão O2:** O checkout reutiliza `AssinarPlanoService` (mód. 19) e `OnboardingFlow` (mód. 21). Nenhum novo comando de domínio é criado — apenas um orquestrador na Application Layer.

## 12. ADRs Necessários

### ADR-012-001: IPaymentGateway (Bloco C)

| Campo | Valor |
|-------|-------|
| Contexto | BillingSimulator simula cobrança. Para produção, é necessário gateway real. |
| Decisão | Criar interface `IPaymentGateway` com métodos charge, cancel, refund. BillingSimulator é substituído por adapter que implementa a interface. Nenhuma entidade de domínio é alterada. |
| Consequências | AssinarPlanoService continua funcionando sem alterações. Gateway pode ser Stripe, Asaas ou qualquer provedor que implemente IPaymentGateway. |

### ADR-012-002: Checkout como Orquestração (Bloco C)

| Campo | Valor |
|-------|-------|
| Contexto | Onboarding + Assinatura + Pagamento precisam ser integrados em um fluxo único. |
| Decisão | Criar um `CheckoutOrchestrator` na Application Layer que compõe `OnboardingFlow.completeStep` + `AssinarPlanoService.Execute` + `PaymentGateway.charge`. Nenhum novo comando de domínio. |
| Consequências | Onboarding e Assinatura permanecem independentes. Checkout é apenas coordenação. |
