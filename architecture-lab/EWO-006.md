## Objetivo

Materializar a Onda 1 da PI-008 (Domain Expansion & Business Rules Completion) através da implementação incremental dos 3 módulos prioritários:

1. **Metas (07)** — Novo módulo: regras de negócio, domínio, contratos, adapters e apresentação
2. **Impostos (08)** — Extensão: regras fiscais completas, anexo técnico, contratos e extensão da feature existente
3. **Rebalanceamento (06)** — Extensão: algoritmos completos, anexo técnico, contratos e extensão da feature existente

Esta Engineering Work Order transforma a especificação arquitetural aprovada em uma sequência rastreável de Slices de implementação, preservando integralmente as Frozen Baselines.

## Princípios Obrigatórios

1. **Extensão, Nunca Modificação** — Nenhum arquivo existente das camadas congeladas é alterado. Toda evolução ocorre por adição.
2. **Uma Slice = um incremento completo** — Compilável, testável, commitável e reversível.
3. **Testes dentro da Slice** — Nenhuma Slice é concluída sem seus testes.
4. **Auditoria obrigatória** — Sem auditoria ChatGPT da Slice anterior, não se inicia a próxima.
5. **Ordem técnica > ordem funcional** — BR + Domínio → App + Infra → Presentation.
6. **Fakes primeiro** — Implementações InMemory precedem as concretas (Supabase) para desacoplar testes.
7. **Zero quebra arquitetural** — PI-008 é a única fonte de arquitetura.
8. **Baseline Lock** — PI-008, ER-008 e EWO-006 entram em congelamento desde o Gate de Entrada.

## Estratégia Geral

### Ordem de Dependências

```
BR + Anexo Técnico (documentação)
        ↓
Core Domain (entidades, value objects, serviços)
        ↓
Application Contracts (Commands, Queries, DTOs, Ports)
        ↓
Infrastructure Adapters (Supabase + InMemory)
        ↓
Presentation Feature (components, hooks, viewmodels, tests)
```

Esta ordem se repete para cada módulo, respeitando a sequência: **Metas → Impostos → Rebalanceamento**.

### Dependências entre Módulos

```
07 — Metas (independente, sem dependências de módulos anteriores)
        ↓
08 — Impostos (depende de Operações e Proventos — já implementados)
        ↓
06 — Rebalanceamento (depende de Carteira — já implementada)
```

### Estratégia de Testes

- **Testes unitários (Vitest):** obrigatórios, cobertura >90% do código novo
- **Testes de integração:** quando a Slice toca fronteiras (Ports, Dispatcher)
- **Testes de arquitetura:** verificar Dependency Rule e Architecture Guard para novos módulos
- **Fakes:** implementações InMemory utilizadas desde a Slice de Infra para viabilizar testes sem Supabase
