## 1. Objetivo

Esta Product Interpretation (PI-006) estabelece a arquitetura oficial da Infrastructure Layer do Lio Feliz.

Seu objetivo é definir como os 10 Ports definidos pela Application Layer (PI-005) serão implementados como adaptadores concretos de infraestrutura, garantindo que o domínio permaneça puro, a application layer independente de tecnologia, e o sistema operacional contra serviços reais (Supabase/PostgreSQL, provedores de mercado, notificações).

A Infrastructure Layer implementa os contratos (Ports) que a Application Layer declarou. Ela não contém regras de negócio — essas pertencem exclusivamente ao domínio (PI-004). Ela não expõe interfaces HTTP ou UI — essas responsabilidades pertencem à Presentation Layer.

Esta PI estabelece:

- o diretório `src/infrastructure/` e sua estrutura;
- o mapeamento Port → Adapter;
- a estratégia de persistência (Supabase/PostgreSQL);
- a estratégia de gateways de dados externos;
- a estratégia de Unit of Work e transações;
- a estratégia de publicação de Domain Events;
- a estratégia de notificações;
- a estratégia de fakes para testes;
- o plano de migração do código legado em `src/lib/`.

## 3. Arquitetura

### 3.1 Princípios

1. **Adapters implementam Ports** — Cada Port da Application Layer é implementado por um ou mais Adapters concretos na Infrastructure Layer. Nenhum adapter expõe métodos além do contrato da interface.

2. **Inversão de Dependência** — A Infrastructure Layer depende da Application Layer (importa as interfaces dos Ports), nunca o contrário.

3. **Domínio Puro** — Nenhuma regra de negócio reside na Infrastructure Layer. Regras de negócio pertencem exclusivamente ao Core Domain (PI-004).

4. **Testabilidade** — Cada adapter deve ter implementação Fake (in-memory) para testes da Application Layer, substituindo os mocks inline atuais.

5. **Migração Gradual** — O código legado em `src/lib/` é migrado para a nova estrutura sem quebrar a funcionalidade existente. A migração ocorre por adapter, com fallback para a implementação legada quando necessário.

### 3.3 Dependências Externas

A Infrastructure Layer utilizará as seguintes dependências (já existentes ou a adicionar):

| Dependência                              | Uso                                    | Já existe? |
| ---------------------------------------- | -------------------------------------- | ---------- |
| `@supabase/supabase-js`                  | Cliente PostgreSQL via Supabase        | Sim        |
| `zod`                                    | Validação de schemas de dados externos | Sim        |
| `node:fetch`                             | Chamadas HTTP para APIs externas       | Nativo     |
| `postgres` ou `@neondatabase/serverless` | Pool de conexões (opcional)            | Não        |

Nenhum ORM (Prisma, Drizzle) será introduzido. O projeto utiliza Supabase JS Client diretamente com queries SQL tipadas via `zod` para validação de resultados.

## 4. Port → Adapter Mapping

### 4.1 PortfolioRepository → SupabasePortfolioRepository

```typescript
// Contrato (Port):
ObterPorId(portfolioId: PortfolioId): Promise<Portfolio | null>
Salvar(portfolio: Portfolio): Promise<void>
ObterTodos(usuarioId: string): Promise<Portfolio[]>

// Implementação (Adapter):
// - Mapeia Portfolio (Aggregate Root) para tabela `portfolios`
// - Serializa/deserializa o aggregate com seus eventos
// - A tabela `portfolios` armazena o estado atual do aggregate
```

### 4.2 AssetRepository → SupabaseAssetRepository

```typescript
// Contrato (Port):
ObterPorId(assetId: AssetId): Promise<Asset | null>
ObterPorTicker(ticker: Ticker): Promise<Asset | null>
Salvar(asset: Asset): Promise<void>
Listar(): Promise<Asset[]>

// Implementação (Adapter):
// - Mapeia Asset para tabela `assets`
// - Consultas por ticker e listagem geral
```

### 4.3 ConfigurationRepository → SupabaseConfigurationRepository

```typescript
// Contrato (Port):
ObterEstrategia(usuarioId: string): Promise<EstrategiaConfiguracao | null>
SalvarEstrategia(usuarioId: string, config: EstrategiaConfiguracao): Promise<void>
ObterMetas(usuarioId: string): Promise<MetaFinanceira[]>
SalvarMetas(usuarioId: string, metas: MetaFinanceira[]): Promise<void>

// Implementação (Adapter):
// - Tabelas `estrategias_configuracao` e `metas_financeiras`
// - Operações de upsert (Salvar substitui registro existente)
```

### 4.4 SubscriptionRepository → SupabaseSubscriptionRepository

```typescript
// Contrato (Port):
ObterPlanoAtivo(usuarioId: string): Promise<Assinatura | null>
Salvar(assinatura: Assinatura): Promise<void>
ListarPlanosDisponiveis(): Promise<PlanoDto[]>

// Implementação (Adapter):
// - Tabelas `assinaturas` e `planos`
// - Planos são dados de referência (seeded)
```

### 4.5 ProjectionRepository → SupabaseProjectionRepository

```typescript
// Contrato (Port):
ObterPatrimonio(portfolioId: string): Promise<PatrimonioProjection | null>
ObterPosicao(portfolioId: string, assetId: string): Promise<PosicaoProjection | null>
ObterPosicoes(portfolioId: string): Promise<PosicaoProjection[]>
ObterHistorico(portfolioId: string, periodo: { inicio: Date; fim: Date }): Promise<HistoricoProjection[]>
ObterProventos(portfolioId: string, filtro: { ano?: number; ticker?: string }): Promise<ProventoProjection[]>

// Implementação (Adapter):
// - Projeções materializadas em tabelas/views de leitura
// - `vw_patrimonio`, `vw_posicoes`, `vw_historico`, `vw_proventos`
// - Atualizadas via triggers ou processos de materialização
// - NOTA: Este Port é somente leitura. A materialização ocorre via Domain Events.
```

### 4.6 DataGateway → Múltiplos Gateways

```typescript
// Contrato (Port):
ObterDadosImportacao(origem: string, parametros: ParametrosImportacao): Promise<DadosImportacao>

// Implementações (Adapters):
// - YahooFinanceGateway → Yahoo Finance API (src/lib/yahoo.server.ts)
// - CoingeckoGateway → CoinGecko API (src/lib/coingecko.server.ts)
// - ExchangeRateGateway → AwesomeAPI (src/lib/exchange.server.ts)
// - B3CsvGateway → Importação de arquivos CSV da B3
```

Cada gateway é resolvido por um factory que seleciona o adapter conforme o parâmetro `origem`.

### 4.7 UnitOfWork → SupabaseUnitOfWork

```typescript
// Contrato (Port):
IniciarTransacao(): Promise<void>
Commit(): Promise<void>
Rollback(): Promise<void>

// Implementação (Adapter):
// - Wrapper sobre transação PostgreSQL (BEGIN/COMMIT/ROLLBACK)
// - Utiliza o client Supabase com `supabase.rpc()` ou SQL direto
```

### 4.8 DomainEventPublisher → InProcessEventPublisher

```typescript
// Contrato (Port):
Publicar(event: DomainEvent): Promise<void>
PublicarVarios(events: DomainEvent[]): Promise<void>

// Implementação (Adapter):
// - Publicação in-process: eventos são dispatchados para handlers registrados
// - Em cenário single-server, não há necessidade de fila externa
// - Se necessário no futuro, pode ser substituído por adaptador de fila sem alterar a Application Layer
```

### 4.9 NotificationPort → InAppNotificationAdapter + EmailNotificationAdapter

```typescript
// Contrato (Port):
Notificar(usuarioId: string, titulo: string, mensagem: string): Promise<void>
NotificarEmail(usuarioId: string, assunto: string, corpo: string): Promise<void>

// Implementações (Adapters):
// - InAppNotificationAdapter → Notificações via Supabase (tabela `notificacoes`)
// - EmailNotificationAdapter → Email via serviço de terceiros (a definir)
```

### 4.10 ImportInterpreterPort → ImportInterpreter

```typescript
// Contrato (Port):
InterpretarOperacao(operacao: OperacaoBruta, portfolioId: string, correlationId: string): FinancialEvent

// Implementação (Adapter):
// - Converte dados brutos importados (CSV, API) em eventos de domínio
// - A lógica de interpretação (mapeamento de colunas, tipos, valores) reside aqui
```

## 5. Estratégia de Persistência

### 5.1 Schema do Banco de Dados

As seguintes tabelas serão criadas/mapeadas no Supabase/PostgreSQL:

| Tabela                     | Port                    | Descrição                                         |
| -------------------------- | ----------------------- | ------------------------------------------------- |
| `portfolios`               | PortfolioRepository     | Estado atual dos aggregates Portfolio             |
| `portfolio_events`         | —                       | Event store para rebuild de aggregates (opcional) |
| `assets`                   | AssetRepository         | Catálogo de ativos                                |
| `estrategias_configuracao` | ConfigurationRepository | Estratégias de alocação                           |
| `metas_financeiras`        | ConfigurationRepository | Metas financeiras dos usuários                    |
| `assinaturas`              | SubscriptionRepository  | Assinaturas ativas                                |
| `planos`                   | SubscriptionRepository  | Catálogo de planos disponíveis                    |
| `vw_patrimonio`            | ProjectionRepository    | View materializada de patrimônio                  |
| `vw_posicoes`              | ProjectionRepository    | View materializada de posições                    |
| `vw_historico`             | ProjectionRepository    | View materializada de histórico                   |
| `vw_proventos`             | ProjectionRepository    | View materializada de proventos                   |
| `notificacoes`             | NotificationPort        | Notificações in-app                               |

### 5.2 Serialização de Aggregates

Aggregates (Portfolio) são serializados para JSON e armazenados na tabela `portfolios`. A deserialização reconstrói o aggregate com seu estado atual.

Para suportar Event Sourcing no futuro, `portfolio_events` pode armazenar a sequência de eventos.

## 6. Estratégia de Fakes para Testes

Cada Port terá uma implementação Fake (in-memory) em `src/infrastructure/fakes/`. Estas implementações:

- Armazenam dados em memória (Mapas/Arrays)
- Não dependem de Supabase, rede ou IO
- São compartilhadas entre todos os testes da Application Layer
- Substituem os mocks inline (`vi.fn()`) atuais

Exemplo:

```typescript
// fake-portfolio-repository.ts
export class FakePortfolioRepository implements IPortfolioRepository {
  private portfolios = new Map<string, Portfolio>();

  async ObterPorId(portfolioId: PortfolioId): Promise<Portfolio | null> {
    return this.portfolios.get(portfolioId.value) ?? null;
  }

  async Salvar(portfolio: Portfolio): Promise<void> {
    this.portfolios.set(portfolio.id.value, portfolio);
  }

  async ObterTodos(usuarioId: string): Promise<Portfolio[]> {
    return Array.from(this.portfolios.values()).filter((p) => p.usuarioId === usuarioId);
  }

  reset(): void {
    this.portfolios.clear();
  }
}
```

## 8. Slices de Implementação

A implementação da Infrastructure Layer será dividida em 7 Slices:

| Slice | Descrição                              | Ports                                                         | Dependências                |
| ----- | -------------------------------------- | ------------------------------------------------------------- | --------------------------- |
| S1    | Fakes + Infra Structure                | Todos (fakes)                                                 | Nenhuma                     |
| S2    | Supabase Repositories (CRUD)           | PortfolioRepository, AssetRepository, ConfigurationRepository | Supabase client, Schema SQL |
| S3    | Subscription + Projection Repositories | SubscriptionRepository, ProjectionRepository                  | S2, Schema SQL              |
| S4    | Unit of Work + Event Publisher         | UnitOfWork, DomainEventPublisher                              | S2                          |
| S5    | Data Gateways (externos)               | DataGateway                                                   | S4                          |
| S6    | Notifications + Import Interpreter     | NotificationPort, ImportInterpreterPort                       | S4                          |
| S7    | Migration + Integration Tests          | Todos                                                         | S1-S6                       |

## 9. Riscos e Mitigações

| Risco                                       | Impacto | Probabilidade | Mitigação                                                           |
| ------------------------------------------- | ------- | ------------- | ------------------------------------------------------------------- |
| Esquema de banco mal projetado              | Alto    | Média         | S1 define fakes primeiro; S2 valida schema com testes de integração |
| APIs externas instáveis (Yahoo, CoinGecko)  | Médio   | Alta          | Gateways com fallback e cache; testes com dados mockados            |
| Quebra de compatibilidade com código legado | Médio   | Baixa         | Migração gradual; fallback ativo até migração completa              |
| Performance de projeções materializadas     | Médio   | Baixa         | Views materializadas com refresh programático; monitoramento        |
| Complexidade de transações distribuídas     | Alto    | Baixa         | UoW PostgreSQL (não distribuído); eventos in-process                |

## 10. Fora do Escopo

Os seguintes itens estão explicitamente fora do escopo da PI-006:

1. **Interface HTTP/API** — Rotas, controllers, GraphQL, REST endpoints pertencem à Presentation Layer.
2. **Autenticação/Autorização como middleware** — Já existe em `src/integrations/supabase/auth-middleware.ts`; adaptação para Ports fica para PI futura.
3. **Filas e mensageria distribuída** — Eventos in-process são suficientes para o cenário atual.
4. **Cache distribuído** — Redis/Memcached não são necessários no momento.
5. **Deployment/Infrastructure-as-Code** — Cloudflare deployment config já existe em `vite.config.ts`.
