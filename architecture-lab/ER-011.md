## 5. Não Conformidades

### NC-011-001 (Baixa, O1) — INotificationPort inconsistente entre §1 e §4.2

- **Local:** §1 (linha 28) vs §4.2 (tabela Novos Ports)
- **Descrição:** §1 cita `INotificationPort` como extensão, mas §4.2 não o inclui na tabela de Novos Ports. O port não precisa de extensão — ele já existe com métodos de notificação. Se o módulo 19 for consumi-lo, a tabela §4.2 deve indicar "CONSUMO" (não extensão).
- **Recomendação:** Substituir "extensão" por "consumo" na §1, ou adicionar `INotificationPort` à tabela §4.2 como port consumido (não estendido).

### NC-011-002 (Baixa, O2) — ILearningPathRepository referenciado mas inexistente

- **Local:** §6, R-016
- **Descrição:** R-016 menciona `ILearningPathRepository` como port consumido pelo módulo 21, mas este port não existe. O módulo 17 expõe `IGlossaryRepository`.
- **Recomendação:** Corrigir referência para `IGlossaryRepository` em R-016.

### NC-011-003 (Baixa, O3) — BillingSimulator ausente da tabela de domínio

- **Local:** §6 (R-015) vs §4.1 (tabela Core Domain)
- **Descrição:** R-015 descreve `BillingSimulator` como serviço de domínio, mas a tabela §4.1 lista apenas `AuthorizationService` para o módulo 19.
- **Recomendação:** Adicionar `BillingSimulator` como serviço de domínio do módulo 19 em §4.1, ou consolidar sua responsabilidade no `AuthorizationService`.

### NC-011-004 (Baixa, O4) — Sobreposição de FR-050 entre PI-010 e PI-011

- **Local:** §8 (Rastreabilidade)
- **Descrição:** PI-010 atribui FR-050 ao módulo 18 (Exportação Avançada). PI-011 inicia em FR-050 para o módulo 19 (Assinaturas). O mesmo número de FR cobre requisitos funcionais distintos.
- **Recomendação:** Iniciar os FRs da PI-011 em FR-051, deslocando: módulo 19 → FR-051 a FR-056, módulo 20 → FR-057 a FR-060, módulo 21 → FR-061 a FR-064, módulo 22 → FR-065 a FR-068.

## 6. Recomendações (sem ação corretiva obrigatória)

### REC-011-001 — Registrar Decisões Arquiteturais como ADRs

As decisões O1, O2 e O3 representam escolhas arquiteturais significativas (extensão de ports vs. criação de novos). Registrar cada uma como ADR na EWO-011 garante rastreabilidade para futuras extensões.

### REC-011-002 — Definir Feature Flags para Plan Capabilities

O módulo 19 define níveis de acesso (FREE, BASIC, PREMIUM) via `AccessLevel`. Recomenda-se implementar um sistema de Feature Flags/Plan Capabilities no `AuthorizationService` para que o controle de acesso por plano seja centralizado e não se espalhe por verificações condicionais no código da Presentation.
